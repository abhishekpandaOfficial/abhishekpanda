SECTION 02

LangGraph vs LangChain — The Decision

This is the most important technical choice in the entire project. Get this wrong and you'll fight your framework for months. Here is the honest comparison:

Option A

LangChain

Linear chain: A → B → C → done
Good for simple Q&A or single-tool tasks
Hard to add conditionals ("if flight found, book it, else search hotels")
State management is messy in complex flows
Can't easily run parallel sub-tasks
Hard to add human-in-the-loop approvals mid-flow
Great for prototypes, struggles in production
NOT CHOSEN
✦ Our Choice

LangGraph

State machine: nodes + edges + conditions
Built for multi-step autonomous agents
Native if/else branching ("needs approval? → pause here")
Typed state object persists across entire task
Parallel edges: search flights AND search hotels simultaneously
Human-in-the-loop interrupts built in — perfect for approval gates
Checkpointing: task can pause, resume after you approve
✓ CHOSEN FOR OPENOWL
The key reason LangGraph wins

OpenOwl needs to pause mid-task and wait for your approval before sending an email or making a payment. LangGraph has this built-in as interrupt_before. In LangChain you'd have to hack this from scratch. LangGraph was literally designed for this pattern.

How LangGraph models OpenOwl's brain

# Every task in OpenOwl is a LangGraph StateGraph ┌─────────────────────────────────────────────────────────────┐ │ OPENOWL STATE GRAPH │ └─────────────────────────────────────────────────────────────┘ START ↓ [receive_input] # WhatsApp / Telegram / SMS / Call ↓ [parse_intent] # classify: book / remind / learn / search / transact ↓ [select_model] # OSS first → fallback chain ↓ [check_approval_needed?] ├── NO → [execute_task] → [update_memory] → [send_reply] → END └── YES → [INTERRUPT: ask_user_approval] ↓ (user responds YES/NO) [resume_from_checkpoint] ├── APPROVED → [execute_task] → [send_reply] └── DENIED → [send_cancelled_msg] → END # Parallel sub-agents via fan-out edges: [execute_task] ├──→ [sub_agent_1: search_flights] ├──→ [sub_agent_2: search_hotels] # run simultaneously └──→ [sub_agent_3: check_calendar] ↓ (fan-in: all done) [aggregate_results]
SECTION 03

AI Model Priority Chain — OSS First, Always

OpenOwl never calls a paid API unless absolutely necessary. The priority chain tries free/local models first. Claude and GPT-4o are only invoked if every OSS option fails or is overloaded. This keeps your monthly AI cost at ₹0 for 95% of tasks.

PRIORITY 1 — LOCAL (FREE, PRIVATE, ZERO LATENCY ON GOOD HARDWARE) Ollama runtime → runs Mistral-7B / LLaMA3.1-8B / Phi-3-mini on YOUR machine No internet needed. 100% private. ₹0/month. Runs on 8GB RAM laptop (slow) or server (fast). PRIORITY 2 — OSS CLOUD API (FREE TIERS) Groq API → Mistral-7B / LLaMA3.1-70B at 300 tokens/sec, FREE tier 6000 req/day Together.ai → LLaMA 3.1, Mistral, Qwen2.5, FREE $25 credit Hugging Face → Inference API free tier, all open models PRIORITY 3 — PAID CLOUD FALLBACK (ONLY IF OSS FAILS) Claude 3.5 Haiku → cheapest Anthropic model, $0.00025/1K tokens (very cheap) GPT-4o-mini → OpenAI cheapest, $0.00015/1K tokens PRIORITY 4 — HEAVY TASKS ONLY (VERY RARE) Claude 3.5 Sonnet → complex reasoning, research tasks GPT-4o → only if Claude fails # Model selection logic considers: task_complexity, context_length, # latency_requirement, current_model_load, user_cost_budget
The ₹0 Reality

Groq's free tier alone handles 6,000 requests/day at 300 tokens/sec. For a personal PA, you'll never exceed 200 requests/day. This means your entire AI inference cost is ₹0/month using just Groq + Ollama. The cloud fallback to Claude/GPT only fires for complex tasks (maybe 5% of the time) — and even then, Claude Haiku costs about ₹0.02 per message.

SECTION 07

Approval Guardrails — Safety Built Into Every Flow

OpenOwl has a hard-coded approval gate for any action that has real-world consequences involving money, communication to others, or irreversible actions. This is implemented at the LangGraph interrupt_before level — not just a prompt instruction. The model literally cannot proceed without your approval signal.

What always requires your approval

  • 💸
    Any money transaction Booking flights, ordering food, paying bills, UPI transfers — OpenOwl shows you the exact amount and merchant before doing anything. You reply YES or NO.
  • 📧
    Sending emails to other people Owl drafts the email, shows it to you first. You can edit, approve, or cancel. It never sends without explicit confirmation.
  • 💬
    Sending WhatsApp/SMS to others If you ask "message Rahul about the meeting", Owl shows you the exact message before sending. You approve it.
  • 🗑️
    Deleting anything Files, emails, calendar events — always asks. Never auto-deletes.
  • 📝
    Publishing or posting Social media posts, blog drafts, GitHub commits — always shown for approval first.
  • 🔐
    Account setting changes Password resets, profile updates, subscription changes — always requires confirmation.

How the approval flow works in real conversation

You send the instruction

"Owl, book me IndiGo 6E-234 Mumbai Friday, window seat"

Owl researches and prepares — but PAUSES

LangGraph hits interrupt_before(execute_payment) — task is frozen in checkpoint state

Owl sends you an approval request

"✈️ Ready to book:
Flight: IndiGo 6E-234 | BOM→HYD | Fri 6:00AM
Seat: 14A (Window) | Price: ₹4,285 + ₹340 fees = ₹4,625
Payment: HDFC Visa **4521

Reply YES to confirm or NO to cancel"

You reply YES

LangGraph resumes from checkpoint. Task executes. Booking confirmed.

Owl confirms and follows up

"✅ Booked! PNR: AB1234. Boarding pass sent to your email. Want a cab reminder for 4AM Friday?"

SECTION 08

Open Source Model — Your Guardrails, Your Rules

The OpenOwl Open Source Philosophy

OpenOwl is open source, but not "do whatever you want" open source. Think of it like Linux: the core is free and open, but the brand and principles are protected. Anyone can build on it, nobody can sell it as closed-source or remove the safety guardrails.

Core Engine
GNU AGPL v3
Anyone can use, modify, deploy for personal use. If you build a business using it, your code must also be open source. Prevents companies from taking it closed-source.
Persona / Plugin System
MIT License
Agent personas, tools, and integrations are MIT — anyone can create and sell plugins without restriction. Creates a healthy ecosystem.
OpenOwl Brand & Name
Trademark Protected
The name "OpenOwl" and the owl logo are trademarked by you. Forks must use a different name. Protects community from confusion.
Guardrail System
Non-Removable (AGPL clause)
The approval gates for money/email/messages are hardcoded and cannot be removed or bypassed in any fork. This is your signature safety contribution.

Your guardrails that CANNOT be forked away

Hard-coded safety contract

Every fork of OpenOwl must retain: (1) approval gates for financial transactions, (2) approval gates for sending messages to third parties, (3) the ability for the user to revoke any app access instantly, (4) no secret data collection, and (5) full conversation logs visible to the user. These are written into the license as mandatory conditions.
