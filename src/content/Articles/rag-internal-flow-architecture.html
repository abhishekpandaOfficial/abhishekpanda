What is RAG?

Retrieval-Augmented Generation

RAG solves one of the fundamental limitations of LLMs — static knowledge. By coupling a retrieval system with a generative model, it produces answers grounded in dynamic, real-world information.

Large Language Models are trained on a fixed snapshot of data. They're extraordinary at reasoning, writing, and synthesis — but they cannot know what happened after their training cutoff, and they can hallucinate facts they don't actually have.

RAG (Retrieval-Augmented Generation), first introduced by Meta AI in 2020, addresses this by adding a live retrieval step before generation. Instead of relying on memorized knowledge, the model receives relevant, up-to-date passages as context — then generates a response grounded in those sources.

The architecture has three conceptual phases: Index (encode your knowledge base), Retrieve (find relevant passages for a query), and Generate (produce a response using retrieved context). NotebookLM, Perplexity AI, and most modern enterprise AI copilots are all built on this pattern.

What makes RAG powerful is that retrieval and generation are decoupled — you can update the knowledge base without retraining the model, making the system both accurate and maintainable.

Step by Step

The 6-Stage RAG Pipeline

Every query processed by a RAG system travels through six distinct stages — from natural language input to a fully grounded, cited response.

1
User Prompt
Input Layer

Everything begins with a natural language question. The user types their query into a chat interface — "Explain transformer self-attention", "What was revenue in Q3?", "Summarise this research paper". This raw text is the entry point to the entire pipeline. At this stage, the system also identifies intent, language, and complexity to decide how to handle downstream steps.

🎯 Intent Classification

Input Processing
type UserQuery = {
  text:      string          // "How does attention work?"
  timestamp: Date
  sessionId: string
  intent:    'qa' | 'summary' | 'generation'
}

const query = preprocess(rawInput)  // clean, tokenize, detect
2
Query Processing & Embedding
Encode

The query text is encoded into a dense numerical vector using an embedding model — the same model used to index the knowledge base. This is critical: both documents and queries must live in the same vector space for similarity search to work. Query expansion techniques (like HyDE — Hypothetical Document Embeddings) may be applied to boost recall by generating a hypothetical answer and embedding that instead.

🔢 Query Embedding
🧠 text-embedding-3-large
📐 768 / 1536 Dimensions
🔄 HyDE (optional)
Embedding
// Encode query into same vector space as documents
const qVec = await embed(query.text)
// qVec: Float32Array[1536]  →  [0.021, -0.183, 0.774, ...]

// Optional: HyDE for harder queries
if (query.intent === 'complex') {
  const hypo = await llm.complete(`Answer briefly: ${query.text}`)
  return embed(hypo)  // embed hypothetical answer instead
}
3
Semantic Search over Knowledge Sources
Retrieve

The query vector is used to run an Approximate Nearest Neighbor (ANN) search against the vector database where all document chunks were previously indexed. The database returns the top-K most semantically similar chunks ranked by cosine similarity. Unlike keyword search, this captures meaning — "automobile" matches "car", "neural net" matches "deep learning model", even with zero word overlap.

🔍 ANN Search (ScaNN / Faiss)
📊 Cosine Similarity
🏆 Top-K Retrieval
📄 PDF · DB · Docs · Code · Web · APIs
Vector Search
const results = await vectorDB.search({
  vector: qVec,
  topK:   8,                          // retrieve 8 best chunks
  filter: { source: userNotebook.id }  // scope to user's data
})
// returns: [{ chunk, score: 0.94 }, { chunk, score: 0.87 }, ...]
4
Context Enrichment
Assembly

The retrieved chunks are assembled into a structured context package alongside their source metadata — document name, page number, timestamp, or URL. An optional re-ranking step (using a cross-encoder model) re-scores and reorders chunks based on relevance to the specific query, improving precision beyond what embedding similarity alone can achieve.

🗂️ Chunk Assembly
📎 Source Metadata
♻️ Cross-Encoder Re-ranking
🔗 Deduplication
Context Assembly
const context = {
  question: query.text,
  chunks: results.map((r, i) => ({
    id:       i + 1,
    text:     r.chunk.text,
    source:   r.chunk.metadata.filename,
    page:     r.chunk.metadata.page,
    score:    r.score
  }))
}  // ready for prompt injection
5
Augmented Prompt Construction
Prompt Eng.

The retrieved context is injected into a carefully engineered prompt template. A system instruction constrains the model to answer only using the provided sources, with mandatory inline citations. This grounding constraint is what prevents hallucination — the model is instructed never to use outside knowledge. The final prompt = System Instruction + Retrieved Sources + User Question.

📝 System Prompt
🔒 Grounding Constraints
📌 Citation Markers [1][2]
📏 Token Budget Management
Prompt Template
SYSTEM:  Answer using ONLY the sources below.
         Cite with [Source N]. Do not use external knowledge.
         If the answer is not in sources, say "Not found in sources."

SOURCES: [Source 1]: ${context.chunks[0].text}
         [Source 2]: ${context.chunks[1].text}
         ... (up to 8 sources)

QUESTION: ${query.text}
6
Grounded Response Generated
Generation

The augmented prompt is sent to the LLM (GPT, Gemini, or Claude). With temperature set low for factual tasks, the model synthesises the retrieved passages into a coherent, fluent, source-cited response. The output is post-processed to verify citations and extract structured data if needed. This final response is what the user sees — accurate, grounded, and fully traceable back to original sources.

🤖 LLM Completion
🌡️ Low Temperature (0.1–0.3)
📚 Cited Answer
✅ Post-processing
LLM Generation
const response = await llm.complete({
  model:       "gpt-4o",        // or gemini-1.5-pro / claude-3.5
  temperature: 0.2,             // low = factual, grounded
  max_tokens:  1024,
  messages:    [{ role: 'user', content: fullPrompt }]
})
return postProcess(response, context.chunks) // verify citations

The Reasoning Engine

LLMs that Power RAG

RAG is model-agnostic — any capable LLM can serve as the generation backend. The choice affects quality, cost, latency, and context window size.

🤖
GPT-4o / GPT-4
OpenAI

The most widely deployed RAG backbone. Excellent instruction-following ensures strict citation compliance. 128K context window handles large document sets. Ideal for enterprise and high-stakes deployments.

🌟
Gemini 1.5 Pro
Google DeepMind

Features a 1M+ token context window — the largest available, enabling entire codebases or book-length documents as context. Powers NotebookLM. Best for multi-document synthesis at scale.

🔶
Claude 3.5 Sonnet
Anthropic

200K context window with exceptional grounding reliability. Claude's constitutional training makes it notably honest about the boundaries of source material — refusing to confabulate beyond retrieved context.

The Case for RAG

Why RAG Wins

RAG solves problems that fine-tuning and prompt engineering alone cannot address.

🧠

No Hallucination on Domain Knowledge

By restricting the model to retrieved sources, RAG architecturally prevents confabulation of domain facts. The model can only reference what you've given it — making it safe for legal, medical, and financial applications.

🔄

No Retraining Required

Fine-tuning an LLM costs thousands of dollars and takes days. With RAG, updating the knowledge base is as simple as adding new documents to the vector index. Changes are live in seconds, not weeks.

📎

Full Source Traceability

Every claim in a RAG response can be traced back to a specific chunk in a specific source document. This auditability is essential for regulated industries and enterprise deployments where answers must be verifiable.

⚡

Works with Private Data

RAG runs entirely on your infrastructure with your documents. Unlike fine-tuning which might leak training data, RAG keeps sensitive information as retrievable context — never baked into model weights.
