What Is SOLID?

Coined by Robert C. Martin ("Uncle Bob") and popularized by Michael Feathers, SOLID is a set of five object-oriented design principles that guide architects and developers toward software that is easier to maintain, extend, test, and reason about — especially as systems grow in complexity.

🏛️
For .NET Architects specifically: These principles are not academic rules — they are the practical difference between a codebase that scales with a team of 50 engineers and one that collapses under the weight of its own complexity. Every enterprise .NET application that stands the test of time — whether it's an ASP.NET Core API, a Blazor app, or a microservices platform — applies these principles at its core.

// PRINCIPLE 01 · SRP

Single Responsibility
Principle

"A class should have only one reason to change."

Every class, module, or function in your codebase should do exactly one thing and do it well. If a class is handling business logic, formatting output, and talking to a database all at once — it has multiple reasons to change and will become a source of bugs, merge conflicts, and test nightmares.

🔵 What
A class should encapsulate one cohesive piece of functionality and have only one reason to change — meaning only one actor or stakeholder should cause it to evolve.
🔵 Why
When a class has multiple responsibilities, a change in one area can break unrelated behavior. Tests become complex, deployment risk increases, and team members constantly step on each other's work.
🔵 How
Ask: "Who will ask me to change this class?" If the answer involves more than one team or domain concern, split the class. Group by use-case, not by technical layer alone.
🔵 When
Apply it whenever a class grows beyond ~200 lines, or when you find yourself writing "and" in a class description: "This class processes orders AND sends emails AND logs activity."

❌ Antipattern — The God Class

InvoiceManager.cs — VIOLATION C# 
// ❌ BAD: One class doing everything
public class InvoiceManager
{
    // Reason 1: Business logic (Finance team)
    public decimal CalculateTotal(Order order)
    {
        return order.Items.Sum(i => i.Price * i.Qty);
    }

    // Reason 2: DB access (Data team)
    public void Save(Invoice invoice)
    {
        using var conn = new SqlConnection(_cs);
        // raw SQL here...
    }

    // Reason 3: Email (Comms team)
    public void SendEmail(Invoice invoice)
    {
        var smtp = new SmtpClient();
        smtp.Send(...);
    }

    // Reason 4: Logging (DevOps team)
    public void LogToFile(string msg)
    {
        File.AppendAllText("log.txt", msg);
    }
}
Services/ — SRP Applied C# 
// ✅ GOOD: Each class has ONE reason to change

public class InvoiceService      // Finance team owns this
{
    public Invoice GenerateInvoice(Order order)
    {
        var total = order.Items
            .Sum(i => i.Price * i.Quantity);
        return new Invoice(order.Id, total);
    }
}

public class InvoiceRepository  // Data team owns this
{
    public void Save(Invoice invoice)
        => _db.Invoices.Add(invoice);
}

public class EmailNotifier      // Comms team owns this
{
    public void SendInvoiceEmail(Invoice inv)
        => _emailClient.Send(inv.CustomerEmail, ...);
}

public class InvoiceLogger      // DevOps team owns this
{
    public void Log(string msg)
        => _logger.LogInformation(msg);
}
🏢 Real Enterprise Scenario

E-Commerce Order Processing System

Imagine an e-commerce platform where a single OrderProcessor class handles validation, payment, inventory deduction, confirmation email, PDF receipt generation, and analytics logging. When the finance department asks to change the PDF format, the developer touches the same class that handles inventory — accidentally introducing a stock miscalculation bug. With SRP, the PDF generation lives in InvoicePdfGenerator, inventory in InventoryService, and payments in PaymentProcessor. Now the finance team's request is isolated to one file, one test suite, one deployment — and nothing else is at risk.

Real-World .NET Use Cases

🏦
Banking: Account + Statement
Separate AccountService (business rules) from StatementGenerator (formatting PDF/HTML) and AccountRepository (data access).
🏥
Healthcare: Patient Records
PatientService handles clinical logic. AuditLogger handles HIPAA compliance logging. Separate concerns protect patient data correctness.
🛒
E-Commerce: Checkout Flow
Split CartService, PricingEngine, VoucherService, TaxCalculator — each team deploys independently without breaking the others.
🚀
Microservices: API Controllers
ASP.NET controllers should only handle HTTP concerns. Business logic lives in service classes, data access in repositories — controllers become thin orchestrators.

🚩 SRP Antipatterns to Avoid

👹
God Class / Manager Class
A single class that orchestrates, validates, persists, notifies, and logs all in one. Usually named XyzManager, XyzHandler, or XyzProcessor with 1000+ lines. Split it along the lines of "who would ask me to change this?"
🔧
Utility / Helper Class Dumping Ground
A Helpers.cs or Utils.cs file where unrelated static methods accumulate over years. Each method has a different "reason to change." Break into focused helpers: StringExtensions, DateTimeHelpers, etc.
📋
Fat ASP.NET Controllers
Controllers that contain business logic, SQL queries, and email sending. Controllers should have one job: receive HTTP, delegate, and return HTTP. Everything else belongs in service classes.

// PRINCIPLE 02 · OCP

Open / Closed
Principle

"Open for extension. Closed for modification."

Software entities should be designed so you can add new behavior by writing new code, not by editing existing, tested code. The goal is that adding a new feature should feel like plugging in a new module — not cracking open the engine while it's running.

🟠 What
Design your classes so that adding new types of behavior requires only adding new classes or implementations — not modifying existing ones that are already tested and working in production.
🟠 Why
Every time you edit existing code, you risk introducing regressions. Over time, a codebase full of if/else chains becomes a minefield — impossible to extend safely without a full regression test suite.
🟠 How
Identify variation points (where behavior differs per type/scenario). Extract an abstraction (interface or abstract base class). Create concrete implementations for each variation. New variation = new class.
🟠 When
Apply when you find yourself writing if (type == "A") … else if (type == "B"), or when the product roadmap shows clear variation in behavior (new payment methods, new discount types, new export formats).
// OCP — EXTENSION WITHOUT MODIFICATION
DiscountCalculator.cs — VIOLATION C# 
// ❌ BAD: Every new discount type forces editing this class
public class DiscountCalculator
{
    public decimal Calculate(string type, decimal price)
    {
        if (type == "standard")
            return price * 0.90m;
        else if (type == "premium")
            return price * 0.75m;
        else if (type == "seasonal")
            return price * 0.80m;
        // Every new type = edit this file = regression risk!
        throw new ArgumentException("Unknown type");
    }
}
Discounts/ — OCP Applied C# 
// ✅ GOOD: New discount = new class, nothing breaks
public interface IDiscountStrategy
{
    decimal Apply(decimal price);
}

public class StandardDiscount : IDiscountStrategy
    public decimal Apply(decimal p) => p * 0.90m;

public class PremiumDiscount : IDiscountStrategy
    public decimal Apply(decimal p) => p * 0.75m;

// ✨ New feature — zero edits to existing code:
public class LoyaltyDiscount : IDiscountStrategy
    public decimal Apply(decimal p) => p * 0.65m;

public class PriceCalculator  // Never changes!
{
    private readonly IDiscountStrategy _discount;
    public PriceCalculator(IDiscountStrategy d)
        => _discount = d;
    public decimal Calculate(decimal p) => _discount.Apply(p);
}

ASP.NET Core DI Registration with OCP

Program.cs C# · ASP.NET Core 
// Register all IDiscountStrategy implementations automatically
// Adding a new strategy = create the class, zero registration code changes
builder.Services.AddScoped<IDiscountStrategy, StandardDiscount>();
builder.Services.AddScoped<IDiscountStrategy, PremiumDiscount>();

// Using a factory pattern (even more OCP-friendly):
builder.Services.AddScoped<IDiscountStrategyFactory, DiscountStrategyFactory>();

// Or auto-register via reflection (zero config needed per new strategy):
var strategies = Assembly.GetExecutingAssembly()
    .GetTypes()
    .Where(t => t.IsAssignableTo(typeof(IDiscountStrategy)) && !t.IsInterface);

foreach (var s in strategies)
    builder.Services.AddScoped(typeof(IDiscountStrategy), s);

🚩 OCP Antipatterns

🌿
Switch/If-Else Type Switching
switch(customerType) blocks that grow with every new customer type. Replace with polymorphism — let each type know how to behave, rather than a central class knowing about all types.
🔀
Direct Class Instantiation for Variation
Using new EmailSender(), new SmsSender() in multiple places. Instead, inject INotificationSender and register the correct implementation in your DI container.

Real-World .NET Use Cases

💳
Payment Gateway Plugins
IPaymentGateway with StripeGateway, PayPalGateway, RazorpayGateway. New payment provider = new class, existing checkout code untouched.
📤
Report Export Formats
IReportExporter with PdfExporter, ExcelExporter, CsvExporter. Product asks for XML export — create XmlExporter, register in DI, done.
🔔
Notification Channels
INotificationChannel with Email, SMS, Push, Slack implementations. Each notification type is a new class — the orchestrator never changes.
🛡️
Validation Pipelines
FluentValidation-style: IValidator<T> implementations chain together. New validation rule = new validator class, existing validators are untouched.

// PRINCIPLE 03 · LSP

Liskov Substitution
Principle

"If S is a subtype of T, objects of T may be replaced with objects of S without altering program correctness."

Coined by Barbara Liskov in 1987, this principle says that any code that works with a base class or interface should work just as well with any subclass — without needing to know which subclass it has. Breaking LSP is the most insidious inheritance bug because the code compiles and runs but produces wrong results silently.

🟢 What
Derived classes must honor the contract of their base class. They can extend behavior but must never weaken preconditions, strengthen postconditions, or throw exceptions the base class doesn't throw.
🟢 Why
Polymorphism is only safe if substitution is guaranteed. When a subclass breaks the contract, callers need if (obj is ChildType) checks — defeating the purpose of inheritance entirely.
🟢 How
Before using inheritance, ask: "Can a consumer use the child class in every place that accepts the parent — with zero surprises?" If the answer is no, use composition instead of inheritance.
🟢 When
Critical when building class hierarchies, domain models with inheritance, and especially when overriding methods. If you find yourself throwing NotImplementedException in an override, LSP is violated.
// THE CLASSIC RECTANGLE / SQUARE LSP VIOLATION
Shapes.cs — LSP VIOLATION C# 
// ❌ BAD: Square "IS-A" Rectangle breaks behavior
public class Rectangle
{
    public virtual int Width { get; set; }
    public virtual int Height { get; set; }
    public int Area() => Width * Height;
}

public class Square : Rectangle
{
    // "Fixing" geometry by coupling Width and Height
    public override int Width
    {
        set { base.Width = value; base.Height = value; }
    }
}

// Consumer code — completely reasonable expectation:
void PrintArea(Rectangle r)
{
    r.Width = 4; r.Height = 5;
    // Expects 20, gets 25 with Square — SILENT BUG 💥
    Console.WriteLine(r.Area());
}
Shapes.cs — LSP Fixed C# 
// ✅ GOOD: Use shared abstraction, not inheritance
public interface IShape
{
    int Area();
}

public class Rectangle : IShape
{
    public int Width { get; }
    public int Height { get; }
    public Rectangle(int w, int h) => (Width, Height) = (w, h);
    public int Area() => Width * Height;
}

public class Square : IShape
{
    public int Side { get; }
    public Square(int s) => Side = s;
    public int Area() => Side * Side;
}

// Consumer works perfectly with both — predictable!
void PrintArea(IShape shape)
    => Console.WriteLine(shape.Area()); // Always correct

.NET Real-World: Repository LSP Pattern

Repositories/ReadOnlyRepo.cs C# · Entity Framework Core 
// ✅ LSP: ReadOnlyRepository can replace IRepository in read-only contexts
public interface IRepository<T>
{
    Task<T> GetByIdAsync(int id);
    Task<IEnumerable<T>> GetAllAsync();
    Task AddAsync(T entity);
    Task UpdateAsync(T entity);
    Task DeleteAsync(int id);
}

// ❌ VIOLATION: ReadOnlyRepository throws on write operations
public class ReadOnlyRepository<T> : IRepository<T>
{
    public Task AddAsync(T e) => throw new NotSupportedException(); // BREAKS LSP!
}

// ✅ FIX: Separate interfaces for read and write concerns
public interface IReadRepository<T>
{
    Task<T> GetByIdAsync(int id);
    Task<IEnumerable<T>> GetAllAsync();
}

public interface IWriteRepository<T> : IReadRepository<T>
{
    Task AddAsync(T entity);
    Task UpdateAsync(T entity);
    Task DeleteAsync(int id);
}

// ReadOnlyRepository safely implements IReadRepository only
public class ProductReadRepository : IReadRepository<Product>
{
    public Task<Product> GetByIdAsync(int id) => _db.Products.FindAsync(id);
    public Task<IEnumerable<Product>> GetAllAsync() => _db.Products.ToListAsync();
}
🚀 Microservices Scenario

Bird Hierarchy in a Wildlife Tracking System

A wildlife system has a Bird base class with a Fly() method. Then someone adds Penguin : Bird and throws NotImplementedException from Fly(). Any code that iterates over a list of birds and calls Fly() — perfectly reasonable — crashes at runtime when it hits a Penguin. The LSP-compliant fix is to have IFlyingBird and ISwimmingBird interfaces. Eagle : IFlyingBird, Penguin : ISwimmingBird. Only flying code works with IFlyingBird, so the constraint is type-safe at compile time, not discovered at runtime on production.

🚩 LSP Antipatterns

💥
NotImplementedException in Overrides
The clearest LSP violation. If a subclass cannot implement a method from its base, the inheritance hierarchy is wrong. Use interface segregation or composition instead.
🔍
Type Checking Consumers (is / as)
When consumers write if (animal is Dog dog) to handle subclass-specific behavior, the abstraction is broken. Polymorphism should handle this transparently.
📐
Changing Method Semantics in Subclasses
A Save() method that in the base class persists synchronously but in a subclass queues to a background job without notifying the caller — silent behavioral change that breaks caller expectations.

Real-World .NET Use Cases

🗃️
EF Core: DbContext Inheritance
Derived AppDbContext : DbContext must honor all base DbContext contracts — override OnModelCreating properly without breaking EF's internal pipeline.
🔐
Authentication Handlers
Custom AuthenticationHandler subclasses in ASP.NET Core must preserve base handler behaviors. Breaking the contract causes silent security failures.
📦
Stream Subclasses
MemoryStream, FileStream, NetworkStream all extend Stream and are interchangeable in any method accepting a Stream. Classic LSP in the BCL.
🏗️
Domain Model Hierarchies
Employee → FullTimeEmployee, ContractEmployee — both must correctly implement CalculatePay() without surprising the payroll service that calls it.

// PRINCIPLE 04 · ISP

Interface Segregation
Principle

"Clients should not be forced to depend on interfaces they do not use."

A class should never be forced to implement methods it doesn't need. Fat interfaces — interfaces with too many responsibilities — force implementing classes to carry dead weight. Small, focused, role-specific interfaces keep contracts clean and implementations lean.

🩷 What
Split large interfaces into smaller, role-based ones. Each interface represents a distinct capability. A class implements only the interfaces whose capabilities it actually provides.
🩷 Why
Fat interfaces create tight coupling between unrelated concerns. When IWorker has Work() and Eat(), robots that implement it must implement Eat() with a stub — that's a design smell surfacing as a code smell.
🩷 How
Group methods by "who needs this." If only some implementors need a method, extract it into a separate interface. Use interface composition: IWorker : IWorkable, IEatable when both make sense together.
🩷 When
Apply when you see classes throwing NotImplementedException, or implementing methods that do nothing, or when a small change to an interface forces recompilation of many unrelated classes.
// ISP — FAT INTERFACE vs SEGREGATED INTERFACES
IWorker.cs — FAT INTERFACE C# 
// ❌ BAD: Robot forced to implement Eat() — absurd!
public interface IWorker
{
    void Work();
    void Eat();
    void Sleep();
    void AttendMeeting();
    void FileTaxReturn();
}

public class Robot : IWorker
{
    public void Work() => Console.WriteLine("Working");
    public void Eat() => throw new NotImplementedException();
    public void Sleep() => throw new NotImplementedException();
    public void AttendMeeting() { /* do nothing */ }
    public void FileTaxReturn() { /* do nothing */ }
    // 4 out of 5 methods are waste!
}
Workers/ — ISP Applied C# 
// ✅ GOOD: Lean, role-based interfaces
public interface IWorkable    { void Work(); }
public interface IFeedable    { void Eat(); }
public interface ISleepable   { void Sleep(); }
public interface IMeetingGoer { void AttendMeeting(); }

// Robot only needs IWorkable
public class Robot : IWorkable
{
    public void Work() => Console.WriteLine("Welding at 200 RPM");
}

// Human employee implements all relevant interfaces
public class HumanEmployee : IWorkable, IFeedable, ISleepable, IMeetingGoer
{
    public void Work()           => Console.WriteLine("Coding");
    public void Eat()            => Console.WriteLine("Lunch");
    public void Sleep()          => Console.WriteLine("Sleeping");
    public void AttendMeeting() => Console.WriteLine("On a call");
}
🏗️ .NET Architect Scenario

Reporting Service: CQRS-Aligned Interface Design

In a CQRS-based reporting system, a naive IReportService might have both GenerateReport(), SaveReport(), EmailReport(), ArchiveReport(), and DeleteReport(). Every consumer — whether it's a read-only dashboard or an admin cleanup job — is forced to depend on all of these. ISP splits this into IReportGenerator, IReportSender, IReportArchiver, and IReportDeleter. The dashboard controller depends only on IReportGenerator — it cannot accidentally call DeleteReport(), and its unit tests are tiny, fast, and focused.

🚩 ISP Antipatterns

🐘
The "Swiss Army Knife" Interface
Interfaces with 15+ methods that force implementors to provide stubs or throw exceptions for half of them. If you need a full mock to test one method, the interface is too fat.
🧩
Service Locator Returning One Giant Interface
A service locator returning IApplicationService that has 30 methods covering five different domains — this forces all consumers to reference the entire surface area of the service.

Real-World .NET Use Cases

📊
ASP.NET Middleware Interfaces
IMiddleware has only InvokeAsync — a perfect ISP example from the BCL. Each middleware concern is isolated in its own class.
🔄
MediatR Handler Interfaces
IRequestHandler<TRequest, TResponse> — each handler implements only the Handle method it cares about. The ultimate ISP pattern in .NET CQRS stacks.
🗄️
Repository: Read vs Write Split
CQRS repositories: IProductReadRepository for query side, IProductWriteRepository for command side. Read services never get exposed to write methods.
⚡
SignalR Hub Interfaces
Typed hubs use a client interface (e.g., IChatClient) — consumers only see the methods relevant to their notification context, not the entire hub surface.

// PRINCIPLE 05 · DIP

Dependency Inversion
Principle

"High-level modules should not depend on low-level modules. Both should depend on abstractions."

The most architecturally powerful of the five principles. DIP is the theoretical foundation for Dependency Injection — the backbone of every modern ASP.NET Core application. When high-level business logic doesn't directly instantiate its dependencies, you can swap implementations at will — for testing, for new environments, for new technology — without touching a single line of business code.

🟡 What
High-level modules (business logic) and low-level modules (infrastructure) both depend on shared abstractions (interfaces). High-level code doesn't know or care which concrete class fulfills the contract.
🟡 Why
Without DIP, swapping your SQL database for a NoSQL store, or your SMTP emailer for SendGrid, requires touching business logic. With DIP, it's a one-line DI container change.
🟡 How
Define interfaces in the domain/application layer. Implement them in the infrastructure layer. Register implementations in the DI container. Never use new ConcreteClass() in high-level code.
🟡 When
Always — every production .NET application. Especially critical when: writing unit tests (inject mocks), supporting multiple environments (dev/staging/prod), or building plugin architectures.
// DIP — DEPENDENCY FLOW BEFORE & AFTER
OrderService.cs — DIP VIOLATION C# 
// ❌ BAD: High-level depends on low-level directly
public class OrderService
{
    // Hard-wired to SQL — impossible to unit test!
    private readonly SqlOrderRepository _repo = new();
    private readonly SmtpEmailService _email = new();
    private readonly FileLogger _logger = new();

    public void PlaceOrder(Order order)
    {
        _repo.Save(order);
        _email.Send(order.CustomerEmail, "Confirmed");
        _logger.Log($"Order {order.Id} placed");
    }
    // Want to use MongoDB? Edit this class.
    // Want to use SendGrid? Edit this class.
    // Want to unit test? Set up SQL, SMTP, filesystem. 😱
}
OrderService.cs — DIP Applied C# 
// ✅ GOOD: Depends on abstractions — inject anything!
public class OrderService
{
    private readonly IOrderRepository _repo;
    private readonly IEmailService _email;
    private readonly ILogger<OrderService> _logger;

    public OrderService(            // Constructor injection
        IOrderRepository repo,
        IEmailService email,
        ILogger<OrderService> logger)
    {
        _repo = repo; _email = email; _logger = logger;
    }

    public async Task PlaceOrderAsync(Order order)
    {
        await _repo.SaveAsync(order);
        await _email.SendAsync(order.CustomerEmail, "Confirmed");
        _logger.LogInformation("Order {Id} placed", order.Id);
    }
    // Swap SQL→Mongo: change DI registration only!
    // Unit test: inject mocks — no real DB needed!
}

⚡ Architecture Scenario

Migrating from SQL Server to Cosmos DB

An enterprise e-commerce platform has 50 services all depending on IProductRepository. The architecture team decides to migrate product storage from SQL Server to Cosmos DB for global distribution. Without DIP, this would mean touching every service. With DIP applied from day one, the migration is: write CosmosProductRepository : IProductRepository, add integration tests, then change one line in Program.cs: services.AddScoped<IProductRepository, CosmosProductRepository>(). Every service keeps working identically — zero business logic changes. This is the power of DIP at architectural scale.

🚩 DIP Antipatterns

🔨
"new" Keyword in Business Logic
var logger = new FileLogger() inside a service class is a compile-time lock. It prevents mocking, testing, and swapping. Move all object creation to the composition root (DI container or factory).
🗃️
Service Locator Pattern
ServiceLocator.Get<ILogger>() inside business classes is DIP in disguise — it still creates a hidden dependency on the locator itself and makes dependencies invisible to callers and tests.
🏛️
Infrastructure References in Domain Layer
The domain project referencing EntityFramework, SendGrid, or Newtonsoft.Json namespaces directly. Domain should define interfaces; infrastructure implements them. The dependency arrow points inward, not outward.

Real-World .NET Use Cases

🏗️
Clean Architecture Layers
Domain defines IRepository<T>. Application uses it. Infrastructure implements it. DIP enforces that the outer layer always depends inward — never the reverse.
🧪
Unit Testing with Mocks
Moq, NSubstitute, and FakeItEasy all rely on DIP — you can only mock what is an interface or virtual. DIP is what makes an entire codebase unit-testable.
☁️
Cloud Provider Abstraction
IStorageService implemented by AzureBlobStorage and AwsS3Storage. Switch cloud providers by changing one DI registration — business code never changes.
🔌
Plugin Architectures
Extensible systems load plugin assemblies and resolve IPlugin implementations at runtime via MEF or custom factories — only possible because everything depends on abstractions.

SOLID Together — The Full Picture

The real power of SOLID emerges when all five principles work in concert. They are not independent rules — they reinforce each other in a coherent architectural philosophy.

💡 Why SOLID Works Together

🔗
SRP + OCP: SRP gives you small, focused classes. OCP ensures those small classes extend without modification — together they create a codebase where every new feature is a new file, not a risky edit.

LSP + ISP: LSP ensures your inheritance hierarchies are safe. ISP ensures your interfaces are slim enough that LSP is never at risk — you can't violate LSP in a single-method interface.

DIP ties it all together: Once you have SRP classes with OCP-compliant designs, ISP-segregated interfaces, and LSP-safe hierarchies — DIP connects them all loosely through your DI container. The result is a system where every piece can be replaced, tested, and extended independently.

All 5 Principles — One Complete .NET Example

OrderProcessing — All SOLID Principles Applied C# · .NET 8 
// ── ISP: Small, role-based interfaces ───────────────────────
public interface IOrderRepository { Task SaveAsync(Order o); }
public interface IOrderReader     { Task<Order> GetAsync(int id); }
public interface INotifier        { Task NotifyAsync(string to, string msg); }

// ── OCP: Extensible notification strategies ──────────────────
public class EmailNotifier : INotifier
    { public Task NotifyAsync(string to, string msg) => /* SMTP */ Task.CompletedTask; }

public class SmsNotifier : INotifier       // New channel — existing code untouched (OCP)
    { public Task NotifyAsync(string to, string msg) => /* Twilio */ Task.CompletedTask; }

// ── LSP: Concrete repository substitutable for interface ─────
public class SqlOrderRepository : IOrderRepository, IOrderReader
{
    public Task SaveAsync(Order o)  => _db.Orders.AddAsync(o).AsTask();
    public Task<Order> GetAsync(int id) => _db.Orders.FindAsync(id).AsTask();
}

// ── SRP: One class, one reason to change ────────────────────
public class OrderPricingService         // Only knows pricing
{
    public decimal CalculateTotal(Order o) => o.Items.Sum(i => i.Price * i.Qty);
}

// ── DIP: High-level depends on abstractions only ─────────────
public class OrderService               // Business logic — knows nothing about SQL/SMTP
{
    private readonly IOrderRepository _repo;
    private readonly INotifier _notifier;
    private readonly OrderPricingService _pricing;

    public OrderService(IOrderRepository repo, INotifier notifier, OrderPricingService pricing)
        => (_repo, _notifier, _pricing) = (repo, notifier, pricing);

    public async Task PlaceAsync(Order order)
    {
        order.Total = _pricing.CalculateTotal(order);
        await _repo.SaveAsync(order);
        await _notifier.NotifyAsync(order.CustomerEmail, $"Order {order.Id} confirmed!");
    }
}

// ── Composition Root (DI) ────────────────────────────────────
builder.Services.AddScoped<IOrderRepository, SqlOrderRepository>();
builder.Services.AddScoped<INotifier, EmailNotifier>();          // Swap to SmsNotifier freely
builder.Services.AddScoped<OrderPricingService>();
builder.Services.AddScoped<OrderService>();
