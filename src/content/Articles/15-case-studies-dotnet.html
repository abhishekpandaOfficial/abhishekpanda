Case Study 01
How WhatsApp Works
2 billion users, 100 billion messages/day, 50 engineers. WhatsApp is one of the most efficient distributed systems ever built — achieved with Erlang (backend) and deliberate simplicity.
2B+ Users100B msgs/day99.999% uptime~50 engineersWebSocket connections
Core Architecture

Connection Layer

  • Each user holds a persistent WebSocket/XMPP connection to a gateway server
  • Connection servers are stateful — they know which users are connected
  • A DHT (Distributed Hash Table) maps UserID → ConnectionServer
  • Message delivery: Sender → Sender's Gateway → DHT lookup → Recipient's Gateway → Recipient
Storage Strategy

Message Store & Delivery

  • Messages stored in Mnesia (Erlang DB) on gateway servers temporarily
  • On delivery ACK received, message deleted — WhatsApp stores as little as possible
  • Media goes to separate blob store (similar to S3), not through message pipeline
  • End-to-end encryption: Signal Protocol — server never sees plaintext
  • Offline queue: messages held until device reconnects
WhatsApp.Gateway/Hubs/ChatHub.cs — .NET SignalR GatewayC# / .NET 9
// WebSocket connection gateway — each user maintains one persistent connection
public class ChatHub : Hub
{
    private readonly IConnectionRegistry _registry;
    private readonly IMessageRouter _router;
    private readonly IOfflineQueueService _offlineQueue;

    public override async Task OnConnectedAsync()
    {
        var userId = Context.User.FindFirst("sub")!.Value;
        // Register: userId → this SignalR connectionId → this server instance
        await _registry.RegisterAsync(userId, Context.ConnectionId);

        // Drain offline queue — messages received while user was away
        var pending = await _offlineQueue.DequeueAllAsync(userId);
        foreach (var msg in pending)
            await Clients.Caller.SendAsync("msg", msg);

        await base.OnConnectedAsync();
    }

    public async Task SendMessage(SendMessageDto dto)
    {
        var senderId = Context.User.FindFirst("sub")!.Value;
        // Message is ALREADY E2E encrypted by the client — we just route it
        var envelope = new MessageEnvelope(
            Id: Guid.NewGuid(), From: senderId,
            To: dto.RecipientId, EncryptedPayload: dto.EncryptedBytes,
            SentAt: DateTimeOffset.UtcNow);

        await _router.RouteAsync(envelope);     // DHT lookup + deliver
        await Clients.Caller.SendAsync("ack", envelope.Id);  // 1 tick ✓
    }

    public override async Task OnDisconnectedAsync(Exception? ex)
    {
        var userId = Context.User.FindFirst("sub")!.Value;
        await _registry.UnregisterAsync(userId);
        await base.OnDisconnectedAsync(ex);
    }
}

// Message router — finds recipient's gateway server via Redis Pub/Sub
public class MessageRouter(IConnectionRegistry registry, IHubContext<ChatHub> hub,
    ISubscriber redis, IOfflineQueueService offlineQueue) : IMessageRouter
{
    public async Task RouteAsync(MessageEnvelope msg)
    {
        var connId = await registry.FindConnectionAsync(msg.To);
        if (connId is not null)
        {
            await hub.Clients.Client(connId).SendAsync("msg", msg);
            // If recipient on DIFFERENT server, publish to Redis channel
        }
        else
        {
            // User offline — queue message for later delivery
            await offlineQueue.EnqueueAsync(msg.To, msg);
        }
    }
}

// Audio streaming service — generates pre-signed URLs and manages playback sessions
public class AudioStreamingService(IBlobStorage storage, ITrackRepository tracks,
    IPublishEndpoint bus, IUserSessionService sessions) : IAudioStreamingService
{
    public async Task<StreamSessionDto> StartStreamAsync(StartStreamRequest req, string userId)
    {
        var track = await tracks.GetByIdAsync(req.TrackId)
            ?? throw new TrackNotFoundException(req.TrackId);

        // Choose quality based on user plan + network quality hint
        var quality = SelectQuality(req.NetworkQuality, req.UserPlan);
        var blobKey = $"tracks/{track.Id}/{quality}.ogg";

        // Generate pre-signed CDN URL valid for 1 hour
        var streamUrl = await storage.GeneratePresignedUrlAsync(blobKey, TimeSpan.FromHours(1));

        var session = new PlaybackSession(userId, track.Id, streamUrl, quality, DateTimeOffset.UtcNow);
        await sessions.SaveAsync(session);

        // Publish play event to Kafka → feeds recommendation engine + royalty calculation
        await bus.Publish(new TrackPlayStartedEvent(userId, track.Id, track.ArtistId, quality, DateTimeOffset.UtcNow));

        return new StreamSessionDto(session.Id, streamUrl, track.DurationMs, GetNextTrackHint(userId, req));
    }

    public async Task ReportPlaybackEventAsync(PlaybackEventDto evt, string userId)
    {
        // Client reports: position every 30s, skip, seek, like
        await bus.Publish(new PlaybackProgressEvent(userId, evt.TrackId, evt.PositionMs,
            evt.EventType, evt.NetworkKbps, DateTimeOffset.UtcNow));

        if (evt.EventType == PlaybackEventType.Skip && evt.PositionMs < 30000)
            // Early skip = negative signal for recommendation model
            await bus.Publish(new EarlySkipEvent(userId, evt.TrackId, evt.PositionMs));
    }

    private static AudioQuality SelectQuality(NetworkQuality net, UserPlan plan) => (net, plan) switch
    {
        (_, UserPlan.Free) => AudioQuality.Low,        // 96kbps
        (NetworkQuality.Poor, _) => AudioQuality.Medium, // 160kbps
        _ => AudioQuality.High                           // 320kbps
    };
}

Case Study 03
How Twitter Timeline Works
The Twitter timeline is one of the most complex feed systems ever built. The core challenge: delivering personalized timelines to 400M users where 50M of them follow over 1000 accounts each — in under 100ms.
400M Users500M Tweets/day<100ms timeline loadPush + Pull hybridRedis timeline cache
Fan-Out Problem

Push vs Pull — The Core Trade-off

  • Pull model: On timeline load, fetch latest tweets from all followed accounts and merge. Simple but slow for high follower counts (Lady Gaga follows 1000 people → 1000 DB queries)
  • Push model (fan-out on write): When a tweet is posted, push it to all followers' cached timelines immediately. Fast reads but explosion for celebrities (100M followers × 1 tweet = 100M writes)
  • Twitter's hybrid: Push for normal users (<1M followers), pull for celebrities, merge at read time
Storage Architecture

What Goes Where

  • Tweet content: MySQL (sharded by tweet ID) + Manhattan (Twitter's distributed store)
  • Timeline cache: Redis sorted sets (score = tweet timestamp) — last 800 tweets per user
  • Social graph: Flock (Twitter's in-house social graph store, similar to Memcached)
  • Media: CDN-backed blob storage (photos/videos)
  • Search index: Earlybird (custom Lucene, real-time indexing within 5 seconds of post)
Twitter.Timeline/Services/TimelineService.cs — Hybrid Fan-OutC# / .NET 9
// Hybrid push/pull timeline — the heart of Twitter's architecture
public class TimelineService(ISocialGraph graph, IDatabase redis,
    ITweetRepository tweets, IPublishEndpoint bus)
{
    private const int CelebFollowerThreshold = 1_000_000;
    private const int MaxTimelineSize = 800;

    // Called when user posts a tweet
    public async Task OnTweetCreatedAsync(Tweet tweet)
    {
        await tweets.SaveAsync(tweet);  // Persist to MySQL

        var followerCount = await graph.GetFollowerCountAsync(tweet.AuthorId);

        if (followerCount < CelebFollowerThreshold)
            // Normal user — fan-out to all follower timelines now (async)
            await bus.Publish(new FanOutTweetCommand(tweet.Id, tweet.AuthorId));
        // Celebrities: no fan-out — their tweets are pulled at read time
    }

    // Fan-out worker — runs asynchronously for normal users
    public async Task FanOutTweetAsync(Guid tweetId, Guid authorId)
    {
        await foreach (var followerBatch in graph.GetFollowersBatchedAsync(authorId, batchSize: 1000))
        {
            var batch = redis.CreateBatch();
            foreach (var followerId in followerBatch)
            {
                var key = $"timeline:{followerId}";
                var score = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds();
                // Add tweet to follower's sorted set (score = timestamp for ordering)
                _ = batch.SortedSetAddAsync(key, tweetId.ToString(), score);
                // Keep only last 800 tweets — trim the tail
                _ = batch.SortedSetRemoveRangeByRankAsync(key, 0, -(MaxTimelineSize + 1));
            }
            await batch.ExecuteAsync();
        }
    }

    // Fetch user's timeline — hybrid pull + cache
    public async Task<IReadOnlyList<Tweet>> GetTimelineAsync(Guid userId, int pageSize = 20)
    {
        // 1. Get tweet IDs from Redis cache (pushed tweets from normal users)
        var cachedIds = await redis.SortedSetRangeByRankAsync($"timeline:{userId}",
            start: 0, stop: pageSize - 1, order: Order.Descending);

        // 2. Find celebrity accounts this user follows — fetch their recent tweets directly
        var celebs = await graph.GetCelebFolloweesAsync(userId, CelebFollowerThreshold);
        var celebTweets = await Task.WhenAll(celebs.Select(c =>
            tweets.GetRecentByAuthorAsync(c, pageSize)));

        // 3. Merge, deduplicate, rank by time, return top N
        var allTweets = await tweets.GetByIdsAsync(cachedIds.Select(id => Guid.Parse(id!)));
        return allTweets
            .Concat(celebTweets.SelectMany(t => t))
            .DistinctBy(t => t.Id)
            .OrderByDescending(t => t.CreatedAt)
            .Take(pageSize)
            .ToList();
    }
}

How a Stock Exchange Works
NASDAQ processes 30 billion orders per day at sub-microsecond latency. The matching engine — the heart of an exchange — must match buy and sell orders with absolute fairness, deterministic ordering, and zero data loss.
30B+ Orders/day<1µs match latencyZero data lossLMAX Disruptor patternCo-location servers
Order Book

The Matching Engine Core

  • Order book: sorted map of Buy orders (descending price) + Sell orders (ascending price)
  • Matching: when highest bid ≥ lowest ask → execute trade at ask price
  • Price-Time Priority (FIFO): same price → earliest order wins
  • All orders processed single-threaded (no locks) — LMAX Disruptor ring buffer pattern
  • Event sourcing: every order event persisted before execution — crash-safe replay
Market Data Feed

Publishing to the World

  • Level 1 data: best bid, best ask, last price — published via UDP multicast to all participants
  • Level 2 data: full order book depth — premium feed for market makers
  • FIX protocol: industry standard for order entry (Financial Information eXchange)
  • Co-location: HFT firms rent rack space in exchange data centers — microseconds of advantage
Exchange.Core/Engine/OrderBook.cs — LMAX Disruptor Pattern in .NETC# / .NET 9
// Order book — the heart of the matching engine. All single-threaded, no locks.
public class OrderBook
{
    // Bids: descending price (highest bid first)
    private readonly SortedDictionary<decimal, Queue<Order>> _bids =
        new(Comparer<decimal>.Create((a, b) => b.CompareTo(a)));
    // Asks: ascending price (lowest ask first)
    private readonly SortedDictionary<decimal, Queue<Order>> _asks = new();

    public IReadOnlyList<TradeExecution> AddOrder(Order incoming)
    {
        var executions = new List<TradeExecution>();
        if (incoming.Side == OrderSide.Buy)
            MatchBuyOrder(incoming, executions);
        else
            MatchSellOrder(incoming, executions);
        return executions;
    }

    private void MatchBuyOrder(Order buy, List<TradeExecution> execs)
    {
        while (buy.RemainingQty > 0 && _asks.Count > 0)
        {
            var (bestAskPrice, askQueue) = _asks.First();
            if (buy.Price < bestAskPrice) break; // No match — bid < lowest ask

            var sell = askQueue.Peek();
            var qty = Math.Min(buy.RemainingQty, sell.RemainingQty);
            execs.Add(new TradeExecution(buy.Id, sell.Id, bestAskPrice, qty, DateTime.UtcNow));

            buy.RemainingQty -= qty;
            sell.RemainingQty -= qty;
            if (sell.RemainingQty == 0) { askQueue.Dequeue(); if (!askQueue.Any()) _asks.Remove(bestAskPrice); }
        }
        if (buy.RemainingQty > 0) AddToBook(_bids, buy); // Rest goes to order book
    }

    private void AddToBook(SortedDictionary<decimal, Queue<Order>> book, Order order)
    {
        if (!book.TryGetValue(order.Price, out var q)) { q = new(); book[order.Price] = q; }
        q.Enqueue(order);
    }
}

// LMAX Disruptor-style sequencer — lock-free ring buffer for order intake
public class OrderSequencer
{
    private readonly Channel<OrderCommand> _channel =
        Channel.CreateBounded<OrderCommand>(new BoundedChannelOptions(1_000_000)
        { SingleReader = true, SingleWriter = false, FullMode = BoundedChannelFullMode.Wait });
    private long _sequence = 0;

    public async Task SubmitAsync(OrderCommand cmd)
    {
        cmd.SequenceNumber = Interlocked.Increment(ref _sequence);
        await _channel.Writer.WriteAsync(cmd);
    }

    // Single reader — matching engine processes one order at a time, no locks needed
    public async IAsyncEnumerable<OrderCommand> ReadAllAsync(CancellationToken ct)
    {
        await foreach (var cmd in _channel.Reader.ReadAllAsync(ct))
            yield return cmd;
    }
}

How Reddit Works
1.5 billion posts, 50 billion votes, 100K+ active communities. Reddit's feed ranking algorithm (Wilson Score + Hot) and vote processing system handles millions of concurrent interactions with eventual consistency at its core.
1.5B+ Posts50B+ Votes100K+ SubredditsHot ranking algorithmPostgreSQL + Cassandra
Vote System

Counting Votes at Scale

  • Votes are written to Kafka — eventual consistency (scores lag by seconds)
  • Vote counting in Cassandra counters (atomic increment/decrement)
  • Anti-vote-manipulation: rate limiting, account age checks, IP detection
  • Score fuzzing: Reddit deliberately obscures exact vote counts by ±10% to prevent brigading detection
Hot Ranking Algorithm

Wilson Score + Time Decay

  • Hot score = log10(ups - downs) + (seconds since epoch - 1134028003) / 45000
  • Time decay: older posts naturally fall — even viral posts sink after hours
  • Wilson Score Lower Bound: for "Top" sort — statistically valid ranking with confidence intervals
  • Scores pre-computed on vote events, cached in Redis sorted sets per subreddit
Reddit.Ranking/Services/HotScoreService.cs — Reddit's Hot AlgorithmC# / .NET 9
public static class RedditRanking
{
    private static readonly DateTimeOffset Epoch = new(2005, 12, 8, 7, 46, 43, TimeSpan.Zero);

    // Reddit's actual hot ranking formula
    public static double HotScore(int ups, int downs, DateTimeOffset created)
    {
        var score = ups - downs;
        var order = Math.Log10(Math.Max(Math.Abs(score), 1));
        var sign = score > 0 ? 1 : score < 0 ? -1 : 0;
        var seconds = (created - Epoch).TotalSeconds;
        return Math.Round(sign * order + seconds / 45000, 7);
    }

    // Wilson Score for "Best" sort — statistically valid with small sample sizes
    public static double WilsonScore(int ups, int total, double z = 1.96)
    {
        if (total == 0) return 0;
        var p = (double)ups / total;
        var n = total;
        return (p + z * z / (2 * n) - z * Math.Sqrt((p * (1 - p) + z * z / (4 * n)) / n))
               / (1 + z * z / n);
    }
}

How a Payment System Works
Stripe processes $1 trillion+ annually. Every payment involves 3–12 hops across banks, card networks, and processors — in under 3 seconds, with exactly-once execution guaranteed. One failed transaction = real money lost.
$1T+ Annually (Stripe)250ms avg latency99.999% uptime SLAPCI DSS Level 1Idempotency-Key critical
Payment Flow

Merchant → Bank → You

  • Authorization: Merchant's payment gateway → Acquirer bank → Card network (Visa/MC) → Issuing bank. Issuer checks: funds, fraud, card valid. Response: Authorized/Declined. ~1-2 seconds.
  • Capture: Merchant requests settlement (usually same day)
  • Settlement: Nets between banks overnight via ACH/SWIFT
  • Refunds: Reverse settlement — issuing bank credits customer
Critical Safety Patterns

Never Charge Twice

  • Idempotency Keys: Every payment request has a unique key — duplicate requests return the same result without re-processing (Stripe's #1 safety feature)
  • Outbox pattern: Payment event and DB write in one transaction — never lose a payment confirmation
  • Saga with compensation: Multi-step payments (auth → reserve → capture) with explicit rollback steps
  • State machine: Payment can only move forward through defined states — no invalid transitions
Payments.Core/Domain/Payment.cs — State Machine + SagaC# / .NET 9
// Payment domain aggregate with state machine
public sealed class Payment : AggregateRoot<PaymentId>
{
    public PaymentStatus Status { get; private set; } = PaymentStatus.Pending;
    public Money Amount { get; private set; }
    public string? AuthorizationCode { get; private set; }
    public string IdempotencyKey { get; private set; }

    public void Authorize(string authCode)
    {
        EnsureStatus(PaymentStatus.Pending);
        AuthorizationCode = authCode;
        Status = PaymentStatus.Authorized;
        RaiseDomainEvent(new PaymentAuthorizedEvent(Id, Amount, authCode));
    }

    public void Capture()
    {
        EnsureStatus(PaymentStatus.Authorized);
        Status = PaymentStatus.Captured;
        RaiseDomainEvent(new PaymentCapturedEvent(Id, Amount));
    }

    public void Fail(string reason)
    {
        if (Status is PaymentStatus.Captured or PaymentStatus.Settled)
            throw new DomainException("Cannot fail an already settled payment");
        Status = PaymentStatus.Failed;
        RaiseDomainEvent(new PaymentFailedEvent(Id, reason));
    }

    private void EnsureStatus(PaymentStatus expected)
    {
        if (Status != expected)
            throw new InvalidPaymentStateException(Id, expected, Status);
    }
}

// Idempotent payment command handler
public class InitiatePaymentHandler(IPaymentRepository repo, IIdempotencyService idempotency,
    IAcquirerGateway acquirer) : IRequestHandler<InitiatePaymentCommand, PaymentResult>
{
    public async Task<PaymentResult> Handle(InitiatePaymentCommand cmd, CancellationToken ct)
    {
        // Check idempotency key — return cached result if duplicate request
        var cached = await idempotency.GetResultAsync<PaymentResult>(cmd.IdempotencyKey);
        if (cached is not null) return cached;

        var payment = Payment.Create(cmd.Amount, cmd.CardToken, cmd.IdempotencyKey);
        await repo.AddAsync(payment, ct);

        var authResult = await acquirer.AuthorizeAsync(payment.Amount, cmd.CardToken, ct);
        if (authResult.IsApproved)
            payment.Authorize(authResult.AuthorizationCode);
        else
            payment.Fail(authResult.DeclineReason);

        await repo.UpdateAsync(payment, ct);  // Outbox pattern saves domain events atomically
        var result = PaymentResult.From(payment);
        await idempotency.StoreResultAsync(cmd.IdempotencyKey, result);
        return result;
    }
}

How Uber Computes ETA
Uber computes billions of ETA predictions daily, powering ride matching, surge pricing, and driver dispatch. Accurate ETAs require real-time traffic, historical patterns, weather, events, and ML-adjusted routing — all in under 500ms.
Billions ETAs/day<500ms per estimateReal-time traffic dataH3 geospatial gridML-corrected routing
ETA Pipeline

From Coordinates to Prediction

  • Map routing: OSRM/Valhalla — Dijkstra on a road graph with weighted edges (speed limits, turn costs)
  • Real-time traffic: GPS pings from active Uber drivers update road segment speeds every 30 seconds
  • ML correction: Historical ETA accuracy data trains a gradient boost model that corrects routing engine estimates
  • H3 geospatial grid: Uber's hexagonal grid divides Earth into cells — demand/supply aggregated per cell
Driver Matching

Finding the Nearest Driver

  • All active drivers stream GPS location every 4 seconds via WebSocket
  • Driver locations stored in Redis Geo (geospatial sorted set by S2 cell)
  • On ride request: GEORADIUS query → candidates ranked by ETA (not raw distance)
  • Dispatch: send offer to top 3 drivers simultaneously, first accept wins
  • Surge pricing: real-time demand/supply ratio per H3 cell

// Real-time driver location tracking — GPS ping every 4 seconds
public class DriverLocationHub(IDatabase redis, IPublishEndpoint bus) : Hub
{
    public async Task UpdateLocation(double lat, double lon, double headingDeg, double speedMph)
    {

        await bus.Publish(new DriverLocationEvent(driverId, lat, lon, speedMph));

// ETA computation — routing + ML correction
public class ETAService(IRoutingEngine routing, IDatabase redis, IMLETAModel model)
{
    public async Task<ETAResult> ComputeETAAsync(GeoPoint origin, GeoPoint destination)
    {
        // 1. Get routing engine estimate (OSRM-like shortest path with traffic weights)
        var route = await routing.ComputeRouteAsync(origin, destination);

        // 2. Apply real-time speed data from Redis (from driver GPS pings)
        var liveSegmentSpeeds = await GetLiveSpeedsAsync(route.Segments, redis);
        var adjustedSeconds = route.Segments.Sum(s =>
            s.LengthMeters / (liveSegmentSpeeds.GetValueOrDefault(s.Id, s.SpeedLimit) * 0.447));

        // 3. ML model correction — trained on historical ETA accuracy
        var features = new ETAFeatures(adjustedSeconds, origin, destination,
            DateTime.UtcNow.DayOfWeek, DateTime.UtcNow.Hour,
            route.DistanceMeters, route.Turns);
        var correctedSeconds = model.Predict(features);

        return new ETAResult(
            EstimatedSeconds: (int)correctedSeconds,
            ConfidenceInterval: model.GetConfidenceInterval(features),
            RoutePolyline: route.Polyline);
    }

    public async Task<IReadOnlyList<NearbyDriver>> FindNearestDriversAsync(GeoPoint pickup, int count = 5)
    {
        // Redis GEO search — find drivers within 5km radius
        var nearby = await redis.GeoSearchAsync("drivers:active",
            new GeoSearchBox(pickup.Lon, pickup.Lat, 5, GeoUnit.Kilometers));

        // Rank by ETA (not raw distance) — a nearby driver in traffic may be slower
        var tasks = nearby.Take(20).Select(async d => {
            var driverPos = await GetDriverPositionAsync(d.Member, redis);
            var eta = await ComputeETAAsync(driverPos, pickup);
            return new NearbyDriver(d.Member!, driverPos, eta.EstimatedSeconds);
        });
        return (await Task.WhenAll(tasks)).OrderBy(d => d.ETASeconds).Take(count).ToList();
    }
}

Inverted Index

  • Inverted index: word → list of DocIDs containing that word (with position + TF-IDF score)
  • Sharded across thousands of machines: each shard owns a slice of the index
  • Query: search term → scatter to all index shards → gather + rank → return top 10
  • PageRank: iterative algorithm — page importance = sum of linking page importances / # outbound links

// Inverted index — the core data structure of every search engine
public class InvertedIndex
{
    // term → (docId, tf-idf score, positions)
    private readonly Dictionary<string, List<PostingEntry>> _index = new();

    public void IndexDocument(Document doc)
    {
        var terms = Tokenize(doc.Content);
        var termFreqs = terms.GroupBy(t => t).ToDictionary(g => g.Key, g => g.Count());
        var maxFreq = termFreqs.Values.Max();

        foreach (var (term, freq) in termFreqs)
        {
            var tf = (double)freq / maxFreq;  // Term frequency (normalized)
            if (!_index.TryGetValue(term, out var postings)) { postings = new(); _index[term] = postings; }
            postings.Add(new PostingEntry(doc.Id, tf, doc.PageRank, FindPositions(terms, term)));
        }
    }

    public IEnumerable<SearchResult> Search(string query, int totalDocs)
    {
        var queryTerms = Tokenize(query);
        // Intersect postings for AND semantics, then rank by TF-IDF × PageRank
        return queryTerms
            .Select(t => _index.GetValueOrDefault(t) ?? [])
            .Aggregate((a, b) => a.IntersectBy(b.Select(e => e.DocId), e => e.DocId).ToList())
            .Select(e => new SearchResult(e.DocId, BM25Score(e, queryTerms, totalDocs) * e.PageRank))
            .OrderByDescending(r => r.Score);
    }

    private static double BM25Score(PostingEntry e, List<string> query, int N,
        double k1 = 1.5, double b = 0.75)
    {
        // BM25 — the industry standard relevance ranking function
        var idf = Math.Log((double)N / 1);
        return idf * (e.TF * (k1 + 1)) / (e.TF + k1);
    }
}

// PageRank — iterative convergence algorithm
public class PageRankCalculator
{
    public Dictionary<string, double> Compute(Dictionary<string, List<string>> links,
        int iterations = 100, double d = 0.85)
    {
        var pages = links.Keys.ToHashSet();
        var N = pages.Count;
        var rank = pages.ToDictionary(p => p, _ => 1.0 / N);

        for (int i = 0; i < iterations; i++)
        {
            var newRank = pages.ToDictionary(p => p, _ => (1 - d) / N);
            foreach (var (page, outLinks) in links)
                foreach (var target in outLinks)
                    newRank[target] += d * rank[page] / outLinks.Count;  // Distribute rank equally
            rank = newRank;
        }
        return rank;
    }
}

        await Task.WhenAll(userChannels.Select(c => Groups.AddToGroupAsync(Context.ConnectionId, $"channel:{c}")));

        // Mark presence as online

    {

        // Validate channel membership
        if (!await members.IsMemberAsync(workspaceId, cmd.ChannelId, userId))
            throw new HubException("Not a member of this channel");

        var message = new Message(workspaceId, cmd.ChannelId, userId, cmd.Text,
            cmd.ThreadTs, DateTimeOffset.UtcNow);

        await bus.Publish(new UpdateUnreadCountCommand(cmd.ChannelId, message.Id));
    }

    {
        // Extend presence TTL — called every 30s by client

public class ChatService(Kernel kernel, IDatabase redis, IChatHistoryRepository historyRepo,
    IContentModerator moderator)
{
    public async IAsyncEnumerable<string> StreamResponseAsync(
        string conversationId, string userMessage, [EnumeratorCancellation] CancellationToken ct)
    {
        // 1. Moderate input — check for policy violations before sending to model
        var modResult = await moderator.CheckAsync(userMessage);
        if (modResult.IsFlagged) { yield return "[Content policy violation]"; yield break; }

        // 2. Load conversation history from Redis (sliding window to fit context)
        var history = new ChatHistory();
        history.AddSystemMessage("You are a helpful AI assistant.");
        var previousMessages = await historyRepo.GetRecentAsync(conversationId, maxTokens: 6000);
        foreach (var msg in previousMessages) history.AddMessage(msg.Role, msg.Content);
        history.AddUserMessage(userMessage);

        // 3. Stream from OpenAI — yield each chunk as it arrives
        var chatCompletion = kernel.GetRequiredService<IChatCompletionService>();
        var fullResponse = new StringBuilder();

        await foreach (var chunk in chatCompletion.GetStreamingChatMessageContentsAsync(history,
            executionSettings: new OpenAIPromptExecutionSettings {
                Temperature = 0.7, MaxTokens = 2048,
                ToolCallBehavior = ToolCallBehavior.AutoInvokeKernelFunctions
            }, kernel: kernel, cancellationToken: ct))
        {
            if (chunk.Content is { Length: > 0 })
            {
                fullResponse.Append(chunk.Content);
                yield return chunk.Content;   // Stream each token to client via SSE
            }
        }

        // 4. Save complete exchange to history for next turn
        await historyRepo.AppendAsync(conversationId,
            new ChatMessage(AuthorRole.User, userMessage),
            new ChatMessage(AuthorRole.Assistant, fullResponse.ToString()));
    }
}

Storage Architecture

How S3 Achieves 11 Nines

  • 3-AZ replication: Every object stored across 3 Availability Zones synchronously
  • Erasure coding: Large objects split into 16 chunks + 4 parity chunks — survive 4 chunk failures
  • Consistent hashing: Object key → hash → storage node (add/remove nodes with minimal redistribution)
  • Write path: PUT → metadata service assigns storage nodes → parallel write to 3 nodes → ACK when 2/3 succeed → return 200
  • Read path: GET → metadata lookup → read from nearest replica
Key Design Patterns

Building a Blob Store in .NET

  • Multipart upload: files >100MB split client-side, each part uploaded independently, assembled server-side
  • Pre-signed URLs: time-limited signed URLs for direct client upload/download (bypass your servers)
  • Versioning: S3 keeps all versions — delete is a soft delete (puts a delete marker)
  • Event notifications: S3 → SQS/SNS/Lambda on PUT/DELETE — drives processing pipelines

// S3-like blob store service — consistent hashing + replication
public class ObjectStorageService(IConsistentHashRing ring, IObjectMetadataRepository metadata,
    IReplicationService replication, IPublishEndpoint bus)
{
    private const int ReplicationFactor = 3;

    public async Task<PutObjectResult> PutObjectAsync(string bucket, string key,
        Stream data, string contentType, CancellationToken ct)
    {
        var objectKey = $"{bucket}/{key}";
        var etag = await ComputeETagAsync(data);

        // Consistent hash → find N storage nodes for this object
        var primaryNode = ring.GetNode(objectKey);
        var replicaNodes = ring.GetReplicaNodes(objectKey, ReplicationFactor - 1);
        var allNodes = new[] { primaryNode }.Concat(replicaNodes).ToArray();

        // Write to all 3 nodes in parallel (quorum write: succeed if 2/3 succeed)
        var writeResults = await Task.WhenAll(allNodes.Select(node =>
            replication.WriteToNodeAsync(node, objectKey, data, ct)));

        var successCount = writeResults.Count(r => r.Success);
        if (successCount < 2) throw new StorageException("Write quorum not met");

        // Write metadata (after successful storage)
        var obj = new StoredObject(objectKey, etag, data.Length, contentType,
            allNodes.Select(n => n.NodeId).ToList(), DateTimeOffset.UtcNow);
        await metadata.SaveAsync(obj, ct);

        // Publish event for downstream processing
        await bus.Publish(new ObjectCreatedEvent(bucket, key, etag, data.Length, contentType));
        return new PutObjectResult(etag, DateTimeOffset.UtcNow);
    }

    public async Task<string> GeneratePresignedUrlAsync(string bucket, string key,
        PresignedUrlOperation operation, TimeSpan validity)
    {
        var expiry = DateTimeOffset.UtcNow.Add(validity).ToUnixTimeSeconds();
        var payload = $"{operation}:{bucket}/{key}:{expiry}";
        var signature = SignHmac256(payload);
        return $"https://storage.example.com/{bucket}/{key}?X-Op={operation}&X-Expires={expiry}&X-Sig={signature}";
    }
}

Creating the Short Code

// Snowflake ID generator — globally unique, time-ordered, no coordination
public class SnowflakeIdGenerator
{
    private static readonly long Epoch = new DateTimeOffset(2024, 1, 1, 0, 0, 0, TimeSpan.Zero).ToUnixTimeMilliseconds();
    private readonly long _workerId;
    private long _sequence = 0, _lastTimestamp = -1;

    public long NextId()
    {
        lock (this) {
            var ts = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds() - Epoch;
            if (ts == _lastTimestamp) { if (++_sequence > 4095) Thread.Sleep(1); }
            else { _sequence = 0; _lastTimestamp = ts; }
            return (ts << 22) | (_workerId << 12) | _sequence;
        }
    }
}

public class UrlShortenerService(SnowflakeIdGenerator idGen, HybridCache cache,
    IUrlRepository repo, IPublishEndpoint bus)
{
    private const string Base62Chars = "0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz";

    public async Task<string> ShortenAsync(string longUrl, string? customCode = null)
    {
        var shortCode = customCode ?? ToBase62(idGen.NextId());
        await repo.SaveAsync(new ShortUrl(shortCode, longUrl, DateTimeOffset.UtcNow));
        return $"https://short.ly/{shortCode}";
    }

    public async Task<RedirectResult> ResolveAsync(string shortCode, HttpContext ctx)
    {
        // HybridCache: L1 in-process + L2 Redis — 99%+ cache hit for hot URLs
        var longUrl = await cache.GetOrCreateAsync($"url:{shortCode}", async _ =>
            (await repo.GetAsync(shortCode))?.LongUrl,
            new HybridCacheEntryOptions { Expiration = TimeSpan.FromHours(24) });

        if (longUrl is null) return RedirectResult.NotFound;

        // Async analytics — don't slow down the redirect for tracking
        _ = bus.Publish(new ClickEvent(shortCode, ctx.Connection.RemoteIpAddress?.ToString(),
            ctx.Request.Headers.Referer, ctx.Request.Headers.UserAgent, DateTimeOffset.UtcNow));

        return RedirectResult.MovedPermanently(longUrl);
    }

    private static string ToBase62(long n)
    {
        var sb = new StringBuilder();
        while (n > 0) { sb.Insert(0, Base62Chars[(int)(n % 62)]); n /= 62; }
        return sb.ToString().PadLeft(7, '0');
    }

How Apache Kafka Works
Kafka processes 7 trillion events per day at LinkedIn. It's not just a message queue — it's a distributed commit log that enables event streaming, event sourcing, and real-time data pipelines at planetary scale. Understanding Kafka is mandatory for senior engineers.
7T Events/day (LinkedIn)1M+ Messages/sec/brokerDistributed commit logConsumer group scalingExactly-once semantics
Core Concepts

Topics, Partitions, Offsets

  • Topic: Named stream of records (like a table in a DB). Partitioned for parallelism.
  • Partition: Ordered, immutable log. Each record has an offset (sequential ID). Messages append-only.
  • Broker: Kafka server that stores partitions. 3+ brokers per cluster for fault tolerance.
  • Consumer Group: N consumers sharing a topic. Each partition assigned to ONE consumer — parallelism = partition count
  • Offset commit: Consumer tracks which messages it's processed. Crash → restart from last committed offset
Why Kafka is Fast

Sequential I/O + Zero Copy

  • Append-only writes to sequential disk — 10x faster than random I/O (100MB/s vs 10MB/s on HDD)
  • Zero-copy: kernel sends data from disk page cache to network socket directly (no userspace copy)
  • Batching: producer batches multiple messages → fewer network round trips
  • Compression: GZIP/Snappy/LZ4 per batch — reduces network bandwidth by 5-7x
  • Replication: leader receives writes, followers replicate asynchronously
