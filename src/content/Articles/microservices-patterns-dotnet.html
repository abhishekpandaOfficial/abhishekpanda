🏗️ What Are Microservices & When Should You Use Them?

Microservices is an architectural style where a large application is broken into small, independently deployable services, each owning its own data, communicating over network APIs. Each service is built around a business capability and can be deployed, scaled, and updated independently.

✅ USE MICROSERVICES WHEN
  • → Large team (>15 engineers) needing independent releases
  • → Different services need different tech stacks/scaling
  • → High availability requirements for specific domains
  • → Clear bounded contexts exist in the domain
  • → Regulatory isolation is needed (PCI, HIPAA)
❌ AVOID MICROSERVICES WHEN
  • → Small team (<5 engineers) — ops overhead kills velocity
  • → Domain boundaries are still unclear
  • → You're just starting (start monolith, extract later)
  • → Low traffic — premature distributed complexity
  • → No DevOps/container expertise yet

Chapter One
Decomposition Patterns
How to slice a monolith into well-defined, independently deployable services
Pattern 1.1
Decompose by Business Capability
Split services along what the business does, not how the code is organized
Beginner
▾

What

Identify the business capabilities your organization performs and create one service per capability. A capability is a stable business function — e.g., Order Management, Payment Processing, Customer Management, Inventory.

Why

Services aligned to business capabilities change together, are owned by one team, and have minimal inter-service dependencies. This matches Conway's Law: your system will mirror your org structure.

When

Use when breaking a monolith, starting a new greenfield distributed system, or when teams are organized around business units (not technical layers like frontend/backend/database).

How in .NET

  • Create separate .NET solutions per capability
  • Each gets its own ASP.NET Core Web API project
  • Each has its own EF Core DbContext + database
  • Use domain events for cross-capability communication
Libraries ASP.NET Core 9 EF Core 9 MediatR (domain events)

TL;DR

Map your org chart's business functions (not tech layers) to services. One service per capability. Each service owns its API, its database, and its domain logic. This is the most natural and stable decomposition strategy.

🛒 E-Commerce platforms🏦 Banking systems🏥 Healthcare SaaS📦 Logistics software
Pattern 1.2
Decompose by Subdomain (DDD)
Use Domain-Driven Design bounded contexts as service boundaries
Intermediate
▾

What

Apply Domain-Driven Design: identify Core subdomains (your competitive advantage), Supporting subdomains, and Generic subdomains. Each bounded context becomes a microservice candidate.

Why

Business capabilities can span multiple contexts and become too broad. DDD gives you a language-based model — within one bounded context, the term "Order" means the same thing to devs and domain experts.

When

Use when domain is complex, when the team has domain experts to work with, when a single "capability" is still too large and contains multiple distinct models of the same concept (e.g. "Product" in Catalog vs Warehouse).

How in .NET

  • Model each bounded context with its own C# domain classes
  • Use distinct DbContext per bounded context
  • Define Aggregates, Entities, Value Objects per context
  • Use Domain Events to cross context boundaries
Libraries MediatR (domain events) EF Core (per context DbContext) Ardalis.Specification Ardalis.SmartEnum
Domain/Orders/OrderAggregate.cs
C#
// Bounded Context: Orders Domain — "Product" means line-item here, NOT catalog item
public sealed class Order : AggregateRoot<OrderId>
{
    private readonly List<OrderLine> _lines = [];

    public CustomerId CustomerId { get; private set; }
    public OrderStatus Status { get; private set; }
    public Money Total => Money.Sum(_lines.Select(l => l.Subtotal));

    public static Order Create(CustomerId customerId)
    {
        var order = new Order { CustomerId = customerId, Status = OrderStatus.Draft };
        order.RaiseDomainEvent(new OrderCreatedDomainEvent(order.Id));
        return order;
    }

    public void AddLine(ProductId productId, Quantity qty, Money unitPrice)
    {
        if (Status != OrderStatus.Draft) throw new DomainException("Cannot modify submitted order");
        _lines.Add(OrderLine.Create(productId, qty, unitPrice));
    }
}

TL;DR

DDD bounded contexts = natural microservice boundaries. The word "Product" in Catalog service and Warehouse service are different models — that's fine and expected. Keep ubiquitous language consistent within a context, and use Domain Events to communicate across boundaries.

Pattern 1.3
Strangler Fig Pattern
Gradually migrate a monolith to microservices without big-bang rewrite
Intermediate
▾

What

Place a facade/proxy in front of the monolith. Incrementally extract functionality into new microservices. Route traffic for extracted features to the new service; everything else still hits the monolith. Over time the monolith "withers" and dies.

Why

Big-bang rewrites are dangerous and usually fail. Strangler lets you extract high-value or high-risk features first, ship incrementally, and reduce risk. Business keeps running throughout.

When

The most important migration pattern. Use whenever you have a legacy monolith to migrate. Extract one bounded context at a time, starting with the most business-critical or the most independently scalable piece.

Steps in .NET

  • Put YARP reverse proxy in front of monolith
  • Extract one service (e.g., Payments) to ASP.NET Core
  • Route /api/payments/* to new service in YARP config
  • Repeat until monolith has no routes left
Libraries YARP (proxy/router) ASP.NET Core (new services) Feature Flags (Microsoft.FeatureManagement)
yarp-proxy/appsettings.json
JSON
{
  "ReverseProxy": {
    "Routes": {
      "payments-route": {          // ✅ Extracted to new microservice
        "ClusterId": "payments-cluster",
        "Match": { "Path": "/api/payments/{**catch-all}" }
      },
      "monolith-route": {          // 🕰️ Everything else still hits monolith
        "ClusterId": "monolith-cluster",
        "Match": { "Path": "/{**catch-all}" }
      }
    },
    "Clusters": {
      "payments-cluster": { "Destinations": { "d1": { "Address": "http://payments-svc:5002" }}},
      "monolith-cluster": { "Destinations": { "d1": { "Address": "http://monolith:3000" }}}
    }
  }
}

TL;DR

Never rewrite a monolith all at once. Put YARP in front, extract one service at a time by routing its URL paths to the new microservice. The monolith gradually loses functionality until it's retired. Zero downtime, incremental risk.

Chapter Two
Communication Patterns
How services talk to each other — synchronous, asynchronous, and event-driven
Pattern 2.1
API Gateway Pattern
Single entry point for all clients — routes, aggregates, secures
Beginner
▾

What

A single-entry reverse proxy that all clients call. It handles cross-cutting concerns: auth, rate limiting, request routing, response aggregation, SSL termination, and logging — so individual services don't have to.

Why

Without a gateway, clients must know all service addresses, handle CORS per service, implement auth on every service independently. Gateway centralizes these concerns and shields the internal topology.

When

Always — this is non-negotiable in production microservices. The gateway is the public face of your system. External clients never talk directly to internal services.

Features to Configure

  • Route matching (path, header, method)
  • Auth middleware (JWT validation)
  • Rate limiting per route/client
  • Request/response transformation
  • Load balancing across service instances
Libraries YARP (Microsoft) Ocelot Azure API Management NGINX Ingress (K8s)
Gateway/Program.cs — YARP Setup
C#
var builder = WebApplication.CreateBuilder(args);

builder.Services
  .AddReverseProxy()
  .LoadFromConfig(builder.Configuration.GetSection("ReverseProxy"));

builder.Services.AddRateLimiter(options =>
  options.AddFixedWindowLimiter("global", o => {
    o.PermitLimit = 100; o.Window = TimeSpan.FromMinutes(1);
  }));

var app = builder.Build();
app.UseAuthentication();      // Validate JWT at gateway
app.UseRateLimiter();
app.MapReverseProxy();        // Routes defined in appsettings.json
app.Run();

TL;DR

The API Gateway is your system's front door. Use YARP for self-hosted, Ocelot for simple cases, or Azure APIM for enterprise. It handles auth, routing, rate limiting, and response aggregation so none of your 30 microservices have to.

Pattern 2.2
Backend for Frontend (BFF)
Dedicated backend per client type — mobile gets a different API than web
Intermediate
▾

What

Create a separate API gateway (or aggregation service) per client type. Mobile BFF returns compact payloads optimized for bandwidth. Web BFF returns richer data. Public API BFF is versioned for 3rd parties. Each BFF aggregates calls to downstream services.

Why

A single gateway becomes a bloated compromise serving all clients poorly. Mobile needs different fields, pagination sizes, and aggregation than web. BFF lets each client team own their backend without coupling to other clients.

When

When you have 2+ significantly different client types (iOS/Android vs Web vs third-party). When client teams are siloed and need independent release cycles. When response shapes differ dramatically between clients.

How in .NET

  • Separate ASP.NET Core project per client type
  • Each BFF calls downstream services via HttpClient
  • Each BFF aggregates/shapes the response for its client
  • Use typed HttpClient + Polly for downstream calls
Libraries ASP.NET Core (per BFF) Polly v8 (resilience) Refit (typed HTTP clients) YARP (within BFF)

TL;DR

Netflix popularized BFF. Instead of one giant API that satisfies no client perfectly, create one slim ASP.NET Core service per client type. Mobile BFF strips down fields; Web BFF joins and enriches. Each BFF team works independently.

Pattern 2.4
Asynchronous Messaging Pattern
Services communicate via messages — decoupled, durable, resilient
Intermediate
▾

What

Services communicate by publishing messages to a broker (Kafka, RabbitMQ, Azure Service Bus). The producer doesn't wait for a response. Consumers process messages at their own pace. Messages are durable — even if the consumer is down, messages queue up.

Why

Synchronous calls create tight coupling — if the downstream service is slow/down, the caller suffers. Async messaging gives you temporal decoupling (services don't need to be up simultaneously) and load leveling.

When

Use for: operations that don't need an immediate response (send email, update analytics), cross-service state changes (order placed → notify inventory/payment), integration with external systems, and any long-running workflows.

How in .NET

  • MassTransit abstracts broker details (Rabbit/Kafka/SB)
  • Define IConsumer<T> for message handling
  • Publish via IBus or IPublishEndpoint
  • Configure saga state machines for workflows
Libraries MassTransit v8 RabbitMQ Apache Kafka (Confluent) Azure Service Bus Dapr pub/sub
Orders/Events/OrderPlacedConsumer.cs — MassTransit
C#
// Publisher — in Orders Service
public class PlaceOrderCommandHandler(IPublishEndpoint bus) : IRequestHandler<PlaceOrderCommand>
{
    public async Task<Unit> Handle(PlaceOrderCommand cmd, CancellationToken ct)
    {
        var order = Order.Create(cmd.CustomerId);
        // ... persist order ...
        await bus.Publish(new OrderPlacedEvent(order.Id, order.Total), ct);
        return Unit.Value;
    }
}

// Consumer — in Notifications Service (separate process)
public class OrderPlacedConsumer : IConsumer<OrderPlacedEvent>
{
    public async Task Consume(ConsumeContext<OrderPlacedEvent> ctx)
    {
        await _emailSender.SendOrderConfirmationAsync(ctx.Message.OrderId);
        await ctx.Publish(new EmailSentEvent(ctx.Message.OrderId));
    }
}

TL;DR

Stop making synchronous calls for everything. Publish events with MassTransit, consume them in separate services. Services become temporally decoupled — if Notifications is down, the order still succeeds and the notification queues up for when it recovers.

Pattern 2.5
Event-Driven Architecture
Services react to events — zero knowledge of who publishes or consumes
Intermediate
▾

What

Services publish domain events that describe what happened ("OrderPlaced", "PaymentProcessed"). Other services subscribe and react. The publisher has zero knowledge of consumers. New services can be added without touching the publisher.

Why

Maximum loose coupling. Adding a new feature (e.g. loyalty points when order placed) requires zero changes to Orders service — just add a new consumer. Also enables event replay, audit logs, and reactive architectures.

When

Business workflows spanning multiple services, audit/compliance requirements, real-time analytics, fan-out scenarios (one event → many consumers), and event sourcing architectures.

Design Rules

  • Events are past-tense facts: "OrderPlaced", not "PlaceOrder"
  • Events are immutable — never change published events
  • Include event version in schema for backwards compat
  • Use idempotent consumers (handle duplicate delivery)
Libraries Apache Kafka (high-throughput) MassTransit (abstraction) Azure Event Grid Dapr pub/sub

TL;DR

Think of events as announcements: "Hey, something happened." Anyone who cares can listen. No one who announces cares who hears. This is the purest form of decoupling — adding the loyalty-points feature never requires touching the Orders service.

Pattern 2.6
Choreography vs Orchestration
Two ways to coordinate multi-step distributed workflows
Intermediate
▾

Choreography

Services react to events autonomously. No central coordinator. Orders publishes "OrderPlaced" → Inventory listens and reserves stock → publishes "StockReserved" → Payments listens and charges. Each service knows only its trigger and its output event.

Orchestration

A central coordinator (saga orchestrator / workflow engine) tells each service what to do and waits for responses. The orchestrator knows the full workflow and drives each step explicitly. Easier to understand the whole flow in one place.

When Choreography

Simple flows with 3–4 steps, loose coupling is paramount, adding steps frequently (just add a new consumer), when you don't want a central bottleneck.

When Orchestration

Complex workflows (8+ steps), when you need compensation on failure, when business visibility of workflow state matters, when debugging a distributed event chain becomes painful.

Choreography Libraries MassTransit (event routing) Kafka
Orchestration Libraries MassTransit Sagas Azure Durable Functions Dapr Workflow

TL;DR

Choreography = jazz band (everyone reacts by ear, no conductor). Orchestration = symphony (conductor drives each musician). Start with choreography for simplicity. Switch to orchestration (MassTransit Saga State Machine) when the workflow becomes complex enough that debugging requires a whiteboard.

Chapter Three
Data Management Patterns
Handling data consistency, distributed transactions, and read/write separation across services
Pattern 3.1
Database per Service
Each service owns its own database — the #1 data pattern in microservices
Beginner
▾

What

Each microservice has its own private database that only it can access directly. No service ever queries another service's database. To get data from another service, you call its API or consume its events — never join across databases.

Why

Shared databases create tight schema coupling. If Inventory changes its schema, it breaks Orders' SQL queries. Private databases enable independent deployment, independent scaling, and even choosing different DB technologies per service (polyglot persistence).

When

Always apply this pattern in true microservices. It's the prerequisite to all other data patterns. If two services share a database, they're a distributed monolith — worse than a regular monolith.

Trade-offs

  • ❌ No cross-service SQL joins (use API Composition or CQRS)
  • ❌ Eventual consistency instead of ACID transactions
  • ❌ Data duplication (necessary — events keep copies in sync)
  • ✅ Independent scaling, independent schema changes
Technologies per Service PostgreSQL (transactional) MongoDB (document) Redis (session/cache svc) CosmosDB (global scale) Cassandra (time-series)

TL;DR

Treat each service's DB as a private implementation detail — other services are forbidden from connecting to it. Data sharing happens through APIs and events only. This is the price of microservices independence, and it's worth paying.

Pattern 3.2
Saga Pattern
Manage distributed transactions across services with compensating transactions
Advanced
▾

What

A Saga is a sequence of local transactions. Each step publishes an event triggering the next. If a step fails, previously completed steps are reversed via compensating transactions. No distributed ACID transaction — each service uses its own local transaction.

Why

You can't use 2-phase commit across microservices at scale (too slow, locks resources). Sagas give you eventual consistency through explicit compensation. "BookOrder" saga: Reserve inventory → Charge payment → Confirm order. If payment fails → Release inventory (compensate).

When

Any multi-step business process spanning multiple services that must all succeed or all roll back. E-commerce checkout, bank transfers, travel booking (flight + hotel + car). The de-facto replacement for distributed transactions.

Types

  • Choreography Saga — events trigger each step (no central state)
  • Orchestration Saga — state machine drives steps (MassTransit Saga)
  • Orchestration is easier to reason about for 4+ steps
  • Always define compensation for each step upfront
Libraries MassTransit Saga State Machine Azure Durable Functions Dapr Workflow API NServiceBus Sagas
Sagas/OrderSaga.cs — MassTransit Saga State Machine
C#
public class OrderSaga : MassTransitStateMachine<OrderSagaState>
{
    public State AwaitingPayment { get; private set; }
    public State AwaitingInventory { get; private set; }
    public State Completed { get; private set; }
    public State Failed { get; private set; }

    public OrderSaga()
    {
        InstanceState(x => x.CurrentState);
        Initially(
          When(OrderPlaced)
            .PublishAsync(ctx => ctx.Init<ReserveInventoryCommand>(new { ctx.Message.OrderId }))
            .TransitionTo(AwaitingInventory)
        );
        During(AwaitingInventory,
          When(InventoryReserved)
            .PublishAsync(ctx => ctx.Init<ChargePaymentCommand>(new { ctx.Message.OrderId }))
            .TransitionTo(AwaitingPayment),
          When(InventoryFailed)
            .Then(ctx => ctx.Saga.FailureReason = "Out of stock")
            .TransitionTo(Failed)  // No compensation needed — nothing committed yet
        );
        During(AwaitingPayment,
          When(PaymentCharged)
            .TransitionTo(Completed),
          When(PaymentFailed)
            .PublishAsync(ctx => ctx.Init<ReleaseInventoryCommand>(new { ctx.Message.OrderId }))
            .TransitionTo(Failed)   // ← Compensation! Release the reserved stock
        );
    }
}

TL;DR

Sagas replace distributed ACID transactions. Each step is a local transaction. On failure, compensating transactions undo completed steps. MassTransit's Saga State Machine gives you a visual, testable workflow with automatic state persistence. Always define your compensation actions before writing the happy path.

Pattern 3.3
CQRS — Command Query Responsibility Segregation
Separate read models from write models for independent optimization
Intermediate
▾

What

Commands change state (Create, Update, Delete) and go through the domain model. Queries only read data and bypass the domain model — they use optimized read models (flat DTOs, materialized views, different databases). Two separate stacks in one service.

Why

Read and write workloads have different shapes. Writes need strong consistency and validation. Reads need speed, denormalization, and aggregation. CQRS lets you optimize each independently — use EF Core for writes, raw Dapper for reads, different DB replicas for reads.

When

When read and write models differ significantly. When reads are far more frequent than writes (most apps). When you need to scale reads independently. When combined with Event Sourcing (natural pairing). Not needed for simple CRUD apps.

How in .NET

  • MediatR: ICommand<T> → CommandHandler, IQuery<T> → QueryHandler
  • Commands: domain validation, EF Core writes
  • Queries: Dapper + raw SQL on read replica
  • Pipeline Behaviors for cross-cutting on both sides
Libraries MediatR (command/query dispatch) EF Core (write side) Dapper (read side) FluentValidation (command validation)
CQRS/Orders — Command + Query Example
C#
// WRITE SIDE — Command goes through domain model
public record PlaceOrderCommand(Guid CustomerId, List<OrderLineDto> Lines) : ICommand<Guid>;

internal sealed class PlaceOrderCommandHandler(IOrderRepository repo, IPublishEndpoint bus)
    : IRequestHandler<PlaceOrderCommand, Guid>
{
    public async Task<Guid> Handle(PlaceOrderCommand cmd, CancellationToken ct)
    {
        var order = Order.Create(cmd.CustomerId);   // ← domain model validates
        foreach (var line in cmd.Lines) order.AddLine(line.ProductId, line.Qty, line.Price);
        await repo.AddAsync(order, ct);
        await bus.Publish(new OrderPlacedEvent(order.Id), ct);
        return order.Id;
    }
}

// READ SIDE — Query bypasses domain, uses raw Dapper for speed
public record GetOrderSummaryQuery(Guid OrderId) : IQuery<OrderSummaryDto>;

internal sealed class GetOrderSummaryQueryHandler(IDbConnection db)
    : IRequestHandler<GetOrderSummaryQuery, OrderSummaryDto>
{
    public async Task<OrderSummaryDto> Handle(GetOrderSummaryQuery q, CancellationToken ct)
        => await db.QuerySingleAsync<OrderSummaryDto>(
            "SELECT o.id, c.name, o.total FROM orders o JOIN customers c ON c.id = o.customer_id WHERE o.id = @id",
            new { id = q.OrderId });   // ← raw SQL join — no domain model needed
}

TL;DR

Commands write through your rich domain model (validation, invariants, domain events). Queries read through the fastest path possible (Dapper, read replicas, materialized views). Never use your EF Core entities for API read responses — create flat DTOs for queries. MediatR is the perfect dispatcher for both sides.

Pattern 3.4
Event Sourcing
Store state as a sequence of events — the event log IS the truth
Expert
▾

What

Instead of storing current state (UPDATE orders SET status='Confirmed'), store every event that changed state (OrderPlaced, PaymentReceived, OrderConfirmed). Current state is derived by replaying events. The event log is append-only and immutable.

Why

Complete audit trail for free (no triggers needed). Time travel — reconstruct state at any point in time. Natural event publishing — events are already there. Debug production bugs by replaying events. Temporal queries ("what was the order state on March 1st at 14:32?").

When

Financial systems (banking, trading), compliance-heavy domains (healthcare, legal), high-value workflows needing full audit trails, systems where "how did we get here" matters as much as "where are we now". Complex, but worth it for the right domain.

Challenges

  • Schema evolution of events (always backward-compatible)
  • Querying current state requires projection/snapshots
  • Higher complexity — not for simple CRUD
  • Event store must be append-only and immutable
Libraries Marten (PostgreSQL event store) EventStoreDB Wolverine (event + messaging) CosmosDB (append patterns)

TL;DR

Store events, not state. "Account balance = 1000" becomes [AccountOpened(500), MoneyDeposited(700), MoneyWithdrawn(200)]. Replay events to get current state. Marten on PostgreSQL is the easiest .NET event store. Always combine with CQRS (projections for the read side).

Pattern 3.6
Transactional Outbox Pattern
Guarantee message delivery even if the message broker goes down
Expert
▾

What

Write the domain event to an "outbox" table in the same database transaction as the business data. A background relay process reads from the outbox table and publishes to the message broker. If broker is down, data is safe in DB. No "saved to DB but message lost" race condition.

Why

The "dual write problem": saving to DB succeeds but publishing to Kafka fails → event lost forever → data inconsistency. Or publish first, but DB save fails → orphan event. The outbox pattern atomically links both operations in one DB transaction.

When

Any time you publish a domain event after a database write in a microservice. This should be your default approach for event publishing — the outbox is a critical reliability pattern, not optional in production systems.

How in .NET

  • MassTransit Outbox (EF Core integration) — easiest
  • Write message to outbox table in same EF Core transaction
  • Background worker (MassTransit relay) publishes messages
  • Mark messages as "published" after successful delivery
Libraries MassTransit Outbox (built-in) Quartz.NET (polling scheduler) Wolverine (built-in outbox)
Program.cs — MassTransit Outbox Configuration
C#
builder.Services.AddMassTransit(x =>
{
    x.AddEntityFrameworkOutbox<AppDbContext>(o =>
    {
        o.UsePostgres();         // outbox tables in same Postgres DB
        o.UseBusOutbox();         // relay messages via background service
    });

    x.UsingRabbitMq((ctx, cfg) => cfg.ConfigureEndpoints(ctx));
});

// Now in your handler — this is ONE atomic transaction:
// 1) EF Core saves the Order to DB
// 2) MassTransit writes OrderPlacedEvent to outbox table
// Both in the same BeginTransaction() → Commit(). Zero dual-write risk.

TL;DR

The outbox pattern solves the "dual write" problem. Save your domain event to the same DB transaction as your business data. A relay process publishes it to the broker asynchronously. MassTransit's built-in EF Core outbox makes this a 5-line configuration. Use this in every production service that publishes events.

Chapter Four
Resilience Patterns
Building services that fail gracefully and recover automatically
Pattern 4.1
Circuit Breaker Pattern
Stop hammering a failing service — open the circuit, fail fast
Beginner
▾

What

Wraps remote calls with a state machine: Closed (normal), Open (failing — reject calls immediately), Half-Open (testing recovery). After N failures, circuit opens. After a timeout, half-open allows one test request. On success → closed. On failure → open again.

Why

Without circuit breaker, a slow downstream service causes your threads to pile up waiting → your service becomes slow → callers wait → cascade failure across the entire system. Circuit breaker isolates the failure and fails fast so callers can use fallbacks.

When

Any HTTP call or database call to an external dependency. Wrap every HttpClient with a circuit breaker. Critical for preventing cascade failures in distributed systems.

States

  • Closed — normal operation, requests flow through
  • Open — circuit tripped, reject all calls immediately
  • Half-Open — trial period, test if service recovered
Libraries Polly v8 (Microsoft.Extensions.Resilience) Dapr (built-in resilience)
Infrastructure/Http/ResilientHttpClient.cs — Polly v8
C#
builder.Services.AddHttpClient<IPaymentsClient, PaymentsClient>()
  .AddResilienceHandler("payments-pipeline", pipeline =>
  {
      // Layer 1: Retry — 3 retries with exponential backoff
      pipeline.AddRetry(new HttpRetryStrategyOptions
      {
          MaxRetryAttempts = 3,
          Delay = TimeSpan.FromSeconds(1),
          BackoffType = DelayBackoffType.Exponential,
          UseJitter = true
      });

      // Layer 2: Circuit Breaker — opens after 5 failures in 30s
      pipeline.AddCircuitBreaker(new HttpCircuitBreakerStrategyOptions
      {
          FailureRatio = 0.5,              // 50% failure rate triggers open
          MinimumThroughput = 5,           // Minimum requests before evaluating
          SamplingDuration = TimeSpan.FromSeconds(30),
          BreakDuration = TimeSpan.FromSeconds(15),  // Stay open for 15s
          OnOpened = args => { _logger.LogWarning("Circuit OPEN for payments!"); return ValueTask.CompletedTask; }
      });

      // Layer 3: Timeout — give up after 5 seconds
      pipeline.AddTimeout(TimeSpan.FromSeconds(5));
  });

TL;DR

Polly v8 is built into .NET 8+ via Microsoft.Extensions.Resilience. Stack Timeout + Circuit Breaker + Retry in that order on every HttpClient. The circuit breaker is your cascade failure prevention — when Payments goes down, Orders fails fast instead of hanging for 30 seconds per request.

Pattern 4.4
Bulkhead Pattern
Isolate resource pools so one slow dependency can't exhaust all threads
Intermediate
▾

What

Named after ship bulkheads that isolate hull sections — if one section floods, the ship doesn't sink. In software: limit the concurrent calls to each downstream service to a separate thread/connection pool. A slow downstream can only block its own pool, not your entire service.

Why

Without bulkheads: if Payments is slow and you have 200 threads, all 200 might queue up waiting for Payments — zero threads left for Orders or Inventory. The entire service hangs even though Payments is the only problem. Bulkheads prevent noisy-neighbor failures.

When

When calling multiple downstream services from one service and you need isolation between them. Critical in API gateways and orchestrator services that fan out to many backends. Also limit concurrent DB connections per tenant in SaaS.

How in .NET

  • Polly v8 ConcurrencyLimiter strategy
  • Separate HttpClient instance per downstream service
  • SemaphoreSlim for manual concurrency control
  • RateLimiter middleware for inbound isolation
Libraries Polly v8 (ConcurrencyLimiter) ASP.NET Core RateLimiter

TL;DR

Ship analogy: compartmentalize your thread pools so one slow service can't flood the whole ship. Give each downstream service its own Polly ConcurrencyLimiter strategy. When the Payments pool is full (20 concurrent calls), new Payments calls fail fast — Orders and Inventory continue normally.

Pattern 4.5
Rate Limiting Pattern
Protect services from traffic spikes and abuse with configurable limits
Intermediate
▾

Algorithms

  • Fixed Window — N requests per minute window (simplest, burst-prone)
  • Sliding Window — smoother, counts in rolling time window
  • Token Bucket — bursts allowed up to bucket capacity, then refills steadily
  • Concurrency Limiter — max N concurrent requests at any time

Why

Prevents DDoS, accidental script floods, runaway bots, and free-tier abuse in SaaS. Also protects downstream services from being overwhelmed. A rate-limited response (429) is far better than a cascaded outage.

When

All public APIs (always). Differentiated limits per client tier (free: 100/min, pro: 10K/min). Protect expensive endpoints (AI inference calls). At gateway level and also at individual service level for defense-in-depth.

How in .NET

  • ASP.NET Core RateLimiter middleware (.NET 7+)
  • Policy per endpoint or global
  • Redis-backed distributed rate limiting for multi-instance
  • Per-client limits using client IP or JWT sub claim
Libraries System.Threading.RateLimiting (.NET 7+) Redis + GSSHA (distributed) YARP Rate Limiter middleware
Program.cs — ASP.NET Core Rate Limiting
C#
builder.Services.AddRateLimiter(options =>
{
    options.RejectionStatusCode = 429;
    options.AddSlidingWindowLimiter("api", o => {
        o.PermitLimit = 100;
        o.Window = TimeSpan.FromMinutes(1);
        o.SegmentsPerWindow = 4;   // 15-second segments
    });

    // Per-client policy using JWT subject claim
    options.AddPolicy("per-user", ctx =>
        RateLimitPartition.GetTokenBucketLimiter(
            ctx.User.FindFirst("sub")?.Value ?? ctx.Connection.RemoteIpAddress?.ToString(),
            _ => new TokenBucketRateLimiterOptions {
                TokenLimit = 500, ReplenishmentPeriod = TimeSpan.FromMinutes(1),
                TokensPerPeriod = 100, AutoReplenishment = true
            }));
});

// Apply on endpoint:
app.MapGet("/api/orders", handler).RequireRateLimiting("per-user");

TL;DR

ASP.NET Core has built-in rate limiting since .NET 7 — no third-party library needed for single-instance. For distributed (multi-pod Kubernetes), use Redis-backed rate limiting. Apply sliding window at gateway, per-user token bucket at service level, concurrency limiter on expensive endpoints.

Pattern 4.6
Health Check Pattern
Expose service health so orchestrators can route traffic and restart unhealthy pods
Beginner
▾

Types

  • Liveness — Is the process alive? (K8s restarts if fails)
  • Readiness — Is the service ready to serve traffic? (K8s stops routing if fails)
  • Startup — Has the service finished initializing? (prevents premature kill)

Why

Without health checks, Kubernetes routes traffic to broken pods and doesn't restart crashed services. Health checks are the primary mechanism for Kubernetes to make autonomous healing decisions. Also exposed via /health endpoint for load balancers.

What to Check

  • Database connectivity (can we connect + query?)
  • Message broker (is RabbitMQ/Kafka reachable?)
  • External API dependencies
  • Memory/CPU thresholds
  • Cache availability (Redis)

How in .NET

  • Microsoft.Extensions.Diagnostics.HealthChecks built-in
  • AspNetCore.HealthChecks.* packages for DB checks
  • Expose /health/live and /health/ready endpoints
  • Configure in Kubernetes probes
Libraries Microsoft.Extensions.Diagnostics.HealthChecks AspNetCore.HealthChecks.NpgSql AspNetCore.HealthChecks.Redis AspNetCore.HealthChecks.RabbitMQ
Program.cs — Health Checks with K8s probes
C#
builder.Services.AddHealthChecks()

  .AddRedis(redisConn, name: "redis", tags: ["cache", "ready"])

Client-Side Service Discovery

▾

What

Why

No single point of failure in routing (no central load balancer). Client can implement smart routing (prefer same-zone instances, latency-based). Works well with dynamic scaling where IPs change constantly in Kubernetes.

Non-Kubernetes environments with Consul or custom registries. When clients need sophisticated routing logic. In Kubernetes, server-side discovery (kube-proxy + Services) is usually preferred over client-side.

Libraries Consul + Steeltoe.Discovery K8s DNS (ClusterIP) Dapr service invocation

TL;DR

In Kubernetes: just use ClusterIP Services + DNS (orders-service.default.svc.cluster.local) — server-side discovery is built into K8s. In non-K8s: use Consul with Steeltoe for client-side discovery. Dapr abstracts this entirely with its service invocation building block.

Pattern 5.3
Service Registry

▾

Tools

Chapter Six
Security Patterns
Securing inter-service and client-to-service communication in distributed systems
Pattern 6.1
JWT Token Authentication
Stateless bearer tokens carry identity across service boundaries
Beginner
▾

What

JSON Web Tokens (JWT) are signed, self-contained tokens containing claims (user ID, roles, permissions). Clients include the token in every request's Authorization header. Services verify the signature with a public key — no round-trip to auth server needed per request.

Why

Session-based auth requires all services to share session state (centralized Redis). JWT is stateless — each service validates the token independently using the public key. No shared session store needed. Scales horizontally perfectly.

When

All service-to-client and service-to-service authentication. JWT at gateway for user auth. Internal service-to-service communication uses JWT with a service identity claim ("svc": "orders-service") or mTLS for stronger guarantees.

How in .NET

  • Gateway validates JWT signature with JWKS endpoint
  • Downstream services validate the same JWT (propagate header)
  • Use claims for authorization: User.HasClaim("role", "admin")
  • Short expiry (15min) + refresh tokens for security
Libraries Microsoft.AspNetCore.Authentication.JwtBearer Duende IdentityServer (issuer) Azure AD B2C / Entra ID
Program.cs — JWT Bearer Auth Setup
C#
builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
  .AddJwtBearer(options =>
  {
      options.Authority = "https://your-identity-server";
      options.Audience = "orders-api";
      options.TokenValidationParameters = new()
      {
          ValidateIssuer = true,
          ValidateAudience = true,
          ValidateLifetime = true,
          ClockSkew = TimeSpan.FromSeconds(30)  // Tolerate 30s clock drift
      };
      // Keys fetched automatically from /.well-known/openid-configuration
  });

builder.Services.AddAuthorization(options => {
    options.AddPolicy("admin-only", p => p.RequireClaim("role", "admin"));
    options.AddPolicy("service-to-service", p => p.RequireClaim("scope", "orders.api"));
});

TL;DR

Validate JWTs at the API gateway (so internal services trust the gateway). Each downstream service also validates (defense-in-depth). Use Duende IdentityServer as your token issuer for full control, or Azure Entra ID for managed auth. Never roll your own JWT implementation.

Pattern 6.2
OAuth2 / OpenID Connect
Delegated authorization and federated identity for microservices
Intermediate
▾

What

OAuth2 is the industry standard for delegated authorization ("allow app X to access resource Y on behalf of user Z"). OIDC extends OAuth2 with identity (ID Token). Together they power SSO, social login, third-party API access, and service-to-service client credentials flow.

Key Flows

  • Authorization Code + PKCE — user-facing apps (SPA, Mobile)
  • Client Credentials — service-to-service (no user involved)
  • Device Code — IoT/CLI tools
  • Never use Implicit flow (deprecated/insecure)

When

User authentication (Authorization Code + PKCE), service-to-service auth (Client Credentials with scopes), third-party API access delegation, SSO across multiple services sharing one identity provider.

Service-to-Service Auth

  • Each service has a client_id + client_secret
  • Request access token from IDP via Client Credentials
  • Include token in downstream HTTP calls
  • Scope-based authorization: "payments.read" scope
Libraries Duende IdentityServer (IDP) Azure Entra ID Microsoft.Identity.Web IdentityModel (Duende BFF)

TL;DR

Use Authorization Code + PKCE for all user-facing auth. Use Client Credentials for service-to-service. Duende IdentityServer is the gold standard self-hosted .NET IDP. For cloud: Azure Entra ID (formerly AAD) with Microsoft.Identity.Web. Never build your own OAuth2 server from scratch.

Pattern 6.3
Mutual TLS (mTLS)
Both client and server authenticate with certificates — zero-trust internal network
Expert
▾

What

In normal TLS, only the server presents a certificate (you trust the server). In mTLS, both client AND server present certificates. The server verifies the client's certificate before accepting the connection. No unauthorized service can connect even if it's inside the cluster.

Why

Zero-trust networking: trust nothing inside the network perimeter. mTLS ensures that only known, certified services communicate. Prevents rogue pods, compromised services, or lateral movement after a breach. Required for PCI-DSS, HIPAA compliance in many interpretations.

When

High-security environments (financial, healthcare). Zero-trust architectures. Compliance mandates. When combined with service mesh (Istio/Linkerd), mTLS is automatic with zero application code changes. Not practical without a service mesh in production.

How in .NET

  • Istio / Linkerd inject sidecar — mTLS automatic
  • Manual: configure Kestrel with client cert requirements
  • Certificate rotation: use cert-manager in K8s
  • SPIFFE/SPIRE for workload identity certificates
Tools Istio (automatic mTLS) Linkerd (automatic mTLS) cert-manager (K8s certs) Kestrel (manual client certs)

TL;DR

Don't implement mTLS manually in application code — use a service mesh. Istio or Linkerd automatically inject sidecars that handle mTLS transparently. Your .NET services don't need any code changes. Enable Istio's STRICT PeerAuthentication policy and every inter-service call gets mTLS for free.

Pattern 6.4
Gateway-Level Authentication
Authenticate once at the gateway — internal services trust forwarded identity
Intermediate
▾

What

The API gateway validates the JWT/OAuth token. After validation, it forwards the request to internal services with the identity information in trusted headers (X-User-Id, X-User-Roles) or re-issues an internal JWT signed by the gateway. Internal services trust these headers without re-validating the original token.

Why

Avoids every service implementing auth logic. Reduces network calls (no round-trip to identity server per service). Central audit point for all authenticated requests. However — internal header trust requires that internal network is not accessible externally.

Trade-offs

  • ✅ Simpler internal services (no auth code)
  • ✅ Single auth implementation point
  • ❌ If gateway is bypassed, internal services are open
  • ❌ Defense-in-depth requires services to also validate

Recommendation

  • Validate at gateway (always)
  • Also validate JWT in services (defense-in-depth)
  • Use Network Policies in K8s to block direct service access
  • Services should never trust X-User-* headers — use the JWT
Libraries YARP + JwtBearer middleware Azure APIM (managed) Duende BFF pattern

TL;DR

Validate at gateway AND in each service. Gateway is your first line — use YARP with JwtBearer middleware. Each downstream service also validates the same JWT for defense-in-depth. Use Kubernetes NetworkPolicy to make services unreachable except from the gateway's namespace.

Observability Patterns
Seeing inside your distributed system — tracing, logging, metrics, and alerting
Pattern 7.1
Distributed Tracing
Follow a request across 15 services with a single Trace ID
Intermediate
▾

What

A unique Trace ID is generated at the entry point (gateway) and propagated through every service call via HTTP headers (traceparent). Each service records a Span (operation unit with timing). Spans are linked by the Trace ID. Visualized as a waterfall in Jaeger/Zipkin/Tempo.

Why

When a request goes through 10 services and takes 3 seconds, where is the bottleneck? Without distributed tracing, you'd have 10 separate log files to correlate manually. With tracing, you get a visual waterfall showing exactly which service/operation took how long.

When

Always — every production microservices system needs distributed tracing. Non-negotiable for debugging latency issues. Set up before you go to production, not after you have a mystery slowdown.

How in .NET

  • OpenTelemetry SDK instruments automatically
  • HTTP calls, DB queries, message publishing all auto-traced
  • Export to Jaeger (self-hosted) or Azure Monitor (cloud)
  • Add custom spans for business operations
Libraries OpenTelemetry .NET SDK Jaeger (self-hosted) Grafana Tempo Azure Application Insights
Program.cs — OpenTelemetry Tracing Setup
C#

  .WithTracing(tracing => tracing
    .AddAspNetCoreInstrumentation()   // HTTP requests auto-traced
    .AddHttpClientInstrumentation()    // Outgoing HTTP calls auto-traced

    .AddMassTransitInstrumentation()   // Message publish/consume traced

using var activity = ActivitySource.StartActivity("ProcessPayment");
activity?.SetTag("payment.amount", amount);
activity?.SetTag("payment.currency", currency);

Add OpenTelemetry to every service. It auto-instruments HTTP, EF Core, HttpClient, and MassTransit with zero code. Traces flow automatically via W3C traceparent headers. Export to Jaeger for self-hosted or Grafana Tempo for production. A single Trace ID traces one user request across all 15 services.

Pattern 7.2
Log Aggregation Pattern
Centralize logs from all services — structured, searchable, correlated
Beginner
▾

What

All services emit structured logs (JSON). A log aggregation system (ELK stack, Loki, Seq) collects, indexes, and searches them. Every log line includes the TraceId so logs and traces can be correlated. Search "TraceId = abc123" and see logs from all 10 services for that request.

Why

SSH-ing into pods is not a debugging strategy at scale. Centralized structured logs let you write SQL-like queries: "find all orders that failed payment in the last hour grouped by customer region". Human-readable strings are searchable but not analyzable.

Structured Logging

  • Log JSON, not strings: {"level":"Error","traceId":"abc","orderId":"123"}
  • Enrich with: service name, version, env, TraceId
  • Never log PII or secrets in log messages
  • Use log levels correctly: Debug (dev), Info (events), Warning (recoverable), Error (needs attention)

How in .NET

  • Serilog with structured output sinks
  • Seq for self-hosted searchable logs
  • Loki + Grafana for production scale
  • OpenTelemetry Logs (new standard, auto-correlates with traces)
Libraries Serilog Seq (self-hosted UI) Grafana Loki OpenTelemetry Logs

TL;DR

Switch from Console.WriteLine to Serilog from day 1. Write structured JSON logs enriched with TraceId and service metadata. Send to Seq locally (free, beautiful UI), Loki in production. Always include the OpenTelemetry TraceId in every log line so you can jump between traces and logs.

Pattern 7.4
Correlation ID Pattern
Thread a unique ID through every hop to correlate logs across services
Beginner
▾

What

Generate a unique ID (GUID) at the gateway for each incoming request. Pass it as a header (X-Correlation-Id or the W3C traceparent). Every service logs this ID with every log line. Every outgoing HTTP call and message includes it. Now "grep for correlation ID" finds all logs across all services for one user action.

Why

Even without full distributed tracing, correlation IDs let you follow a request through log files across 10 services. The MVP observability pattern — implement this first, before anything else. Distributed tracing (OpenTelemetry) builds on this concept and supersedes it with richer context.

When

Every microservices system always. This is the minimum viable observability. Implement on day 1. OpenTelemetry's TraceId/SpanId serve this same purpose automatically — use OTel if you're setting up tracing anyway.

How in .NET

  • Middleware extracts/generates X-Correlation-Id header
  • Serilog enricher adds it to all log properties
  • HttpClient DelegatingHandler propagates it outbound
  • AsyncLocalStorage or IHttpContextAccessor for passing through code
Libraries Serilog.Enrichers.CorrelationId Custom ASP.NET Core Middleware OpenTelemetry (supersedes)

TL;DR

The simplest observability pattern. Generate a GUID at entry, thread it through every log, every HTTP header, every message. Costs zero and saves hours of debugging. If you're using OpenTelemetry, the TraceId IS your correlation ID — no need to implement separately.

Chapter Eight
Deployment Patterns
Zero-downtime releases, progressive rollouts, and infrastructure as code
Pattern 8.1
Blue-Green Deployment
Two identical environments — instant switch, instant rollback
Intermediate
▾

What

Run two identical production environments (Blue = current, Green = new version). Deploy new version to Green while Blue serves all traffic. Run smoke tests on Green. Switch load balancer from Blue to Green in seconds. If Green fails, switch back to Blue instantly. Zero downtime, instant rollback.

Why

Traditional deployments require a window ("we'll be down from 2-4am"). Blue-green eliminates that. Rollback from a bad deploy is a load balancer toggle — seconds, not a full redeployment. High confidence deployments.

When

Any service where downtime is unacceptable. In Kubernetes, use with slot-based deployments (Azure App Service slots) or duplicate Deployment manifests with a Service selector switch. For breaking DB schema changes — requires backward-compatible schemas during transition.

DB Challenge

  • Both Blue and Green must work with the same DB schema during transition
  • Use expand-contract migrations: add columns first (backward compat), then remove old ones after Blue is retired
  • Never drop columns in the same release that introduces the feature
Tools Kubernetes (selector switching) Azure App Service Slots GitHub Actions (workflow) YARP (traffic switching)

TL;DR

Two environments, instant traffic switch. In K8s: deploy new Deployment (green), run smoke tests, update Service selector to point to green pods — takes 1 second. Rollback: revert selector. DB migrations must be backward-compatible across both environments. Costlier (double infra) but safest.

Pattern 8.2
Canary Release
Gradually shift traffic to new version — catch bugs before full rollout
Intermediate
▾

What

Deploy new version alongside old, initially receiving 5% of traffic. Monitor error rate and latency. If healthy, increase to 25% → 50% → 100%. If error rate spikes, instantly redirect all traffic back to old version. Named after coal miners using canary birds to detect gas leaks.

Why

Real production traffic finds bugs that staging never replicates. Limiting blast radius to 5% of users while validating a release is far safer than blue-green's all-or-nothing switch. Used by Netflix, Google, Amazon for every production release.

When

Risk-averse releases of complex features. Large user bases where even 1% of failures is significant. When you want real production validation before full commitment. Combine with feature flags for even finer control.

How in K8s

  • Istio VirtualService with traffic weight splitting
  • NGINX Ingress canary annotations
  • KEDA + YARP for weighted routing in .NET
  • Feature flags (Microsoft.FeatureManagement) for code-level canary
Tools Istio (traffic splitting) Argo Rollouts Microsoft.FeatureManagement Grafana (metrics monitoring)

TL;DR

Start canary at 1-5% traffic. Watch error rate and p99 latency in Grafana. Auto-rollback triggers if error rate exceeds threshold (Argo Rollouts does this automatically). Increment to 25/50/100% on success. For code-level canary: feature flags with Microsoft.FeatureManagement give you fine-grained % rollout per feature.

Pattern 8.3
Sidecar Pattern
Attach a helper container to each service pod for cross-cutting concerns
Advanced
▾

What

Each service pod has a secondary "sidecar" container that handles cross-cutting concerns (logging, tracing, mTLS, config refresh, service mesh proxy). The main container (your .NET service) communicates with the sidecar over localhost. The sidecar handles network plumbing.

Why

Cross-cutting infrastructure (service mesh, log forwarding, secrets rotation) shouldn't be implemented in every service's code. Sidecars inject this capability without code changes. The .NET service stays clean and focused on business logic.

Examples

  • Envoy/Istio proxy — handles mTLS, traffic routing, observability
  • Dapr sidecar — pub/sub, state management, service invocation
  • Fluent Bit — log collection and forwarding
  • Secret fetcher — pulls secrets from Vault and mounts them

How with Dapr

  • Dapr injects sidecar automatically via K8s webhook
  • .NET service calls localhost:3500 (Dapr HTTP API)
  • Dapr handles pub/sub, state store, service invocation
  • Zero broker-specific code in .NET service
Tools Dapr (full sidecar platform) Istio/Envoy (service mesh) Fluent Bit (log sidecar)

TL;DR

Sidecar = attach a helper process to each pod that handles plumbing. Dapr is the easiest .NET sidecar: one YAML annotation on your K8s deployment injects a sidecar that gives you pub/sub, state management, distributed tracing, and service invocation — zero broker-specific code in your .NET service.

Pattern 8.4
Service Mesh
Infrastructure layer for service-to-service communication — zero app code
Expert
▾

What

A dedicated infrastructure layer (Istio, Linkerd) that handles all service-to-service networking via sidecar proxies (Envoy). Provides: automatic mTLS, traffic management (canary, blue-green), distributed tracing, retry/circuit breaker policies, and observability — all without a single line of application code.

Why

Implementing retry, circuit breaker, mTLS, and tracing in every service's Polly/HttpClient is duplication at scale. A service mesh handles all of this once at the infrastructure level. Application code becomes pure business logic. Security policies are enforced even if devs forget to implement them.

When

Large organizations (50+ microservices), strict security requirements, when you can't trust devs to implement Polly correctly on every service, when you need traffic management without redeploying services. Istio has real operational complexity — don't use for <10 services.

Trade-offs

  • ✅ Zero app code for resilience, mTLS, tracing
  • ✅ Uniform policies across all services
  • ❌ Operational complexity — steep Istio learning curve
  • ❌ Sidecar adds ~10ms latency per hop and memory overhead
  • ❌ Istio debugging is difficult when things go wrong
Tools Istio (feature-rich, complex) Linkerd (simpler, Rust-based) Dapr (app-level mesh) Azure Service Mesh (managed)

TL;DR

Service mesh is infrastructure-level Polly + mTLS + tracing for free. Linkerd is simpler and more .NET-friendly than Istio. Dapr is a developer-centric alternative that runs as a sidecar and provides similar benefits at application level. Use Linkerd for K8s service mesh; use Dapr if you want app-level building blocks.

Chapter Nine
Cross-Cutting Patterns
Essential patterns that apply across all services — caching, idempotency, and anti-corruption
Pattern 9.1
Idempotency Pattern
Safely retry operations — same request twice = same result, no duplicate effects
Intermediate
▾

What

An operation is idempotent if performing it multiple times has the same effect as performing it once. Clients send an Idempotency-Key header (GUID). Server checks if that key was processed before — if yes, return the cached result. If no, process and store result with the key.

Why

Networks are unreliable. Retry logic (Polly) might retry a "CreateOrder" request if the response times out — even though the first attempt succeeded. Without idempotency, you'd have two orders charged. With idempotency keys, retry is safe.

When

Payment endpoints (critical — charging twice is catastrophic). Order creation. Any non-idempotent write operation that clients might retry. Async message consumers must ALWAYS be idempotent (at-least-once delivery means duplicates are guaranteed).

How in .NET

  • Extract Idempotency-Key from request header
  • Check Redis/DB: has this key been processed?
  • If yes: return stored response (200 OK, same body)
  • If no: process, store response + key, return result
  • Set TTL on stored keys (24-48h)
Libraries Redis (idempotency key store) Custom ASP.NET Core Filter MassTransit (consumer idempotency)
Filters/IdempotencyFilter.cs
C#
public class IdempotencyFilter(IDistributedCache cache) : IActionFilter
{
    public async void OnActionExecuting(ActionExecutingContext ctx)
    {
        if (!ctx.HttpContext.Request.Headers.TryGetValue("Idempotency-Key", out var key)) return;

        var cached = await cache.GetStringAsync(key);
        if (cached is not null)
        {
            // Already processed — return stored response, skip handler
            ctx.Result = new ContentResult { Content = cached, ContentType = "application/json", StatusCode = 200 };
        }
    }

    public async void OnActionExecuted(ActionExecutedContext ctx)
    {
        if (!ctx.HttpContext.Request.Headers.TryGetValue("Idempotency-Key", out var key)) return;
        if (ctx.Result is ObjectResult { Value: not null } result)
        {
            var body = JsonSerializer.Serialize(result.Value);
            await cache.SetStringAsync(key, body, new DistributedCacheEntryOptions
                { AbsoluteExpirationRelativeToNow = TimeSpan.FromHours(24) });
        }
    }
}

TL;DR

Every payment and order endpoint MUST be idempotent. Clients include a UUID Idempotency-Key header. Store key+response in Redis with 24h TTL. Return cached response on duplicate. MassTransit message consumers: always check if the event was already processed before acting. Stripe, PayPal, and every serious payment API uses this.

Pattern 9.2
Cache-Aside Pattern
Application manages cache explicitly — check cache, load DB on miss, populate cache
Beginner
▾

What

Application code checks cache (Redis) first. On hit: return cached data. On miss: load from DB, write to cache with TTL, return data. Application explicitly manages what's in cache. The data store is the source of truth; cache is a performance layer.

Why

Most read-heavy microservices would saturate their DB without caching. 80% of reads typically hit 20% of data. Caching that 20% with Redis reduces DB load by 80%. Essential for product catalog, user profiles, configuration data, and reference data reads.

When

Data that is read frequently and changes infrequently (product catalog, user profile, pricing). Set appropriate TTL based on staleness tolerance. Don't cache data that must always be real-time (account balance, inventory count during checkout).

Cache Invalidation

  • TTL-based: simplest, accept eventual consistency
  • Write-through: update cache on DB write
  • Event-driven: publish "ProductUpdated" event → consumers clear cache
  • Cache versioning: include version in key prefix

// .NET 9 HybridCache — combines in-memory (L1) + Redis (L2)
public class GetProductQueryHandler(HybridCache cache, IProductRepository repo)
    : IRequestHandler<GetProductQuery, ProductDto>
{
    public async Task<ProductDto> Handle(GetProductQuery q, CancellationToken ct)
        => await cache.GetOrCreateAsync(
            key: CacheKeys.Product(q.ProductId),           // "product:guid"
            factory: async _ => await repo.GetByIdAsync(q.ProductId, ct),
            options: new HybridCacheEntryOptions
            {
                Expiration = TimeSpan.FromMinutes(15),         // L2 Redis TTL
                LocalCacheExpiration = TimeSpan.FromMinutes(2) // L1 in-memory TTL
            },

Use .NET 9's HybridCache for the best of both worlds: in-process memory cache (L1, microseconds) + Redis (L2, sub-millisecond). Check → Hit returns instantly. Miss → loads from DB, populates both levels. Invalidate via event: when ProductUpdated published, remove the key from Redis. Default TTL for catalog data: 15 minutes.

Pattern 9.3
Anti-Corruption Layer (ACL)
Translate between external/legacy models and your clean domain model
Advanced
▾

What

A translation layer between your service's domain model and an external system (third-party API, legacy monolith, another bounded context with different vocabulary). The ACL converts the foreign model to your clean domain concepts, preventing external models from corrupting your domain language.

Why

A legacy system calls "Product" what you call "Item" and has a "customerCode" where you have "CustomerId". Without an ACL, their ugly model leaks into your clean code. ACL keeps your domain model pure and localized.

When

Integrating with third-party APIs (Stripe, Salesforce, SAP). Integrating with a legacy monolith during strangler migration. When two bounded contexts have different models of the same concept. Any integration point where external vocabulary differs from your domain language.

How in .NET

  • Create an IExternalSystemGateway interface in Application layer
  • Implement it in Infrastructure layer with the translation logic
  • Domain model never directly depends on external DTOs
  • Use AutoMapper or manual mapping in the ACL class
Libraries Refit (external HTTP client) Custom Gateway/Adapter classes Polly (resilience for external calls)

TL;DR

Define an interface (IPaymentGateway) in your Application layer that speaks your domain language (ProcessPayment(Money amount, CustomerId customer)). Implement it in Infrastructure using Stripe's API. Stripe's WeirdStripeRequest object never enters your domain. Swap Stripe for Razorpay by changing the Infrastructure implementation — domain code unchanged.

Pattern 9.4
Inbox Pattern
Guarantee exactly-once message processing — the consumer-side outbox
Expert
▾

What

The complement to the Outbox pattern, but on the consumer side. When a message arrives, write it to an "inbox" table first (with MessageId) in the same DB transaction as your business operation. Before processing, check if MessageId was already seen. Prevents duplicate processing of the same message.

Why

Message brokers guarantee at-least-once delivery — your consumer WILL receive duplicates (network retries, rebalancing, broker restarts). Without the Inbox pattern, a "ChargePayment" consumer might charge a customer twice when the broker re-delivers a message.

When

Combine with Outbox for full end-to-end reliability guarantee. Essential for payment consumers, order confirmation consumers, and any state-changing message handler. The Outbox prevents lost messages on the publisher; the Inbox prevents duplicate processing on the consumer.

How in .NET

  • MassTransit InMemory or EF Core Inbox (built-in)
  • Store MessageId + ProcessedAt in inbox table
  • Check inbox before processing handler logic
  • Entire operation (inbox write + business logic) in one DB transaction
Libraries MassTransit (built-in Inbox) Wolverine (built-in idempotent delivery)

TL;DR

Outbox (publisher side) + Inbox (consumer side) = reliable messaging. MassTransit provides both out of the box with EF Core integration. Add UseInboxFilter() to your consumer registration. Every message has a MessageId — storing it in your inbox table with the business operation in one transaction makes processing exactly-once safe.
