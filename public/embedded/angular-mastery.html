Chapter 01 — Foundation

Introduction to Angular

Angular is a platform and framework for building client-side applications using HTML and TypeScript. Developed and maintained by Google, it provides a complete solution with everything you need out of the box.

Angular 21 TypeScript-First Google Maintained Component-Based
Why Angular? Concepts
🏗️

Opinionated Architecture

Angular provides a complete, opinionated architecture with a well-defined structure, making large team collaboration predictable and scalable.

🔒

TypeScript First

Built from the ground up with TypeScript, giving you type safety, better IDE support, and early error detection at compile time.

⚡

Powerful CLI

Angular CLI automates scaffolding, building, testing, and deployment — dramatically reducing repetitive setup and configuration work.

🔄

Two-Way Data Binding

NgModel enables seamless two-way data binding between component class and template, keeping UI and data in sync effortlessly.

💉

Dependency Injection

Angular's built-in DI system makes your code more modular, testable, and maintainable by decoupling component logic from service logic.

🧪

Testability

First-class support for unit testing (Jasmine/Jest) and end-to-end testing (Playwright/Cypress) built into the ecosystem.

Angular History & Versions
1

AngularJS (2010)

The original framework using JavaScript, two-way data binding, and MVC architecture. Became hugely popular but had performance bottlenecks at scale.

2

Angular 2 (2016)

Complete rewrite in TypeScript. Introduced components, directives, services, and the modern DI system. Breaking change from AngularJS.

3

Angular 4–8 (2017–2019)

Semantic versioning adopted. Ivy renderer introduced in preview. Differential loading, web workers, and Bazel support added.

4

Angular 9–12 (2020–2021)

Ivy becomes default. Strict mode enabled. Lazy loading improvements, better tree-shaking, and huge bundle size reductions.

5

Angular 13–16 (2022–2023)

Standalone components introduced (no NgModule needed). Signals API preview. esbuild integration. Vite-powered dev server.

6

Angular 17–21 (2024–2025)

New Control Flow syntax (@if, @for, @switch). Signals stable. Deferrable views. SSR with Hydration. Zoneless change detection. @let syntax. Incremental hydration.

Modules & Standalone Components

Angular modules organize the application into cohesive blocks of functionality. Angular 17+ promotes standalone components as the default — but understanding NgModules is essential for maintaining legacy code and large enterprise apps.

NgModule Anatomy

  declarations: [/* components/directives/pipes */],
  imports: [/* other modules */],
  exports: [/* what to expose */],
  providers: [/* services */],
  bootstrap: [AppComponent] // root only

export class AppModule {}

@Component({
  standalone: true, // No NgModule needed!
  imports: [
    CommonModule,
    ReactiveFormsModule,

export class ProductsComponent {}
Feature Module Types
🏢

Domain Feature Module

Contains all UI related to a specific business domain (e.g., Products, Orders). Has its own routing, components, and services scoped to that domain.

⏳

Lazy Loaded Feature Module

Loaded on demand when the user navigates to the feature route. Dramatically reduces initial bundle size. Use loadComponent() for standalone or loadChildren() for modules.

🛣️

Routing Feature Module

Dedicated to setting up RouterModule.forChild() routes for a feature. Keeps routing logic separate from component logic.

🌐

Shared Module

Contains reusable components, directives, and pipes imported by multiple feature modules. Never provide services here — they'll create multiple instances.

🔑

Core Module

Singleton services (AuthService, LoggingService), HTTP interceptors, and application-wide guards. Import ONLY in AppModule once.

⚙️

Root AppModule

The top-level module that bootstraps the application. Imports BrowserModule, CoreModule, and AppRoutingModule. Components declared here are app-wide.

Lazy Loading with Standalone Components
// app.routes.ts — Angular 17+ preferred approach
export const routes: Routes = [
  {
    path: 'dashboard',
    loadComponent: () =>
      import('./features/dashboard/dashboard.component')
        .then(m => m.DashboardComponent)
  },
  {
    path: 'products',
    loadChildren: () =>
      import('./features/products/products.routes')
        .then(m => m.PRODUCT_ROUTES)
  }
];

// main.ts — Standalone bootstrap
bootstrapApplication(AppComponent, {
  providers: [
    provideRouter(routes, withPreloading(PreloadAllModules)),

⚠️ Common Mistake: Never import BrowserModule in feature modules — only in AppModule. In standalone apps, use CommonModule or specific imports. Double-importing BrowserModule causes runtime errors.
Chapter 04 — Core Building Block

Components In Depth

Components are the fundamental building blocks of Angular applications. Each component consists of a TypeScript class, an HTML template, and optional CSS styles — together defining a piece of the UI.

Lifecycle HooksContent ProjectionChange Detection
Component Lifecycle Hooks
1

ngOnChanges()

Called before ngOnInit() and whenever an @Input property changes. Receives a SimpleChanges object with previous and current values. Use for reacting to input changes.

2

ngOnInit()

Called once after the first ngOnChanges(). Best place to fetch data, initialize the component, and subscribe to observables. Runs after constructor.

3

ngDoCheck()

Called during every change detection run. Use to implement custom change detection logic. Runs very frequently — keep it lightweight.

4

ngAfterContentInit() / ngAfterContentChecked()

Called after Angular projects external content into the component's view via <ng-content>. Use @ContentChild safely here.

5

ngAfterViewInit() / ngAfterViewChecked()

Called after the component's view and child views are initialized. Safe to access @ViewChild elements here. Avoid triggering change detection inside these.

6

ngOnDestroy()

Called just before Angular destroys the component. Clean up subscriptions, event listeners, and timers here to prevent memory leaks.

Change Detection Strategies

Default Strategy

Angular checks the entire component tree on every event, setTimeout, HTTP request, and Promise resolution. Simple but can be slow for large apps.

@Component({
  changeDetection: ChangeDetectionStrategy.Default
})

OnPush Strategy ⭐ Recommended

Only re-renders when: input references change, an event originates from this component, async pipe resolves, or manually triggered with markForCheck(). Massive performance win.

@Component({
  changeDetection: ChangeDetectionStrategy.OnPush
})
export class OptimizedComponent {
  constructor(private cdr: ChangeDetectorRef) {}
  refresh() { this.cdr.markForCheck(); }
}
💡 Angular 21 Zoneless: Use provideExperimentalZonelessChangeDetection() in bootstrap to remove Zone.js entirely and rely on Signals for change detection — giving you the fastest Angular apps possible.

Custom Attribute Directive
@Directive({
  selector: '[appHighlight]',
  standalone: true
})
export class HighlightDirective {
  @Input('appHighlight') color = 'yellow';
  @Input() defaultColor = 'transparent';

  constructor(private el: ElementRef, private renderer: Renderer2) {}

  @HostListener('mouseenter')
  onMouseEnter() {
    this.renderer.setStyle(this.el.nativeElement, 'backgroundColor', this.color);
  }

  @HostListener('mouseleave')
  onMouseLeave() {
    this.renderer.setStyle(this.el.nativeElement, 'backgroundColor', this.defaultColor);
  }
}

// Usage: <p [appHighlight]="'cyan'" defaultColor="gray">Hover me</p>
💡 Best Practice: Always use Renderer2 instead of directly accessing nativeElement styles. Renderer2 is platform-safe and works with server-side rendering (SSR).

Custom Pipe with Parameters
@Pipe({
  name: 'truncate',
  standalone: true,
  pure: true // default — only re-runs when input reference changes
})
export class TruncatePipe implements PipeTransform {
  transform(value: string, limit = 50, trail = '...'): string {
    if (!value || value.length <= limit) return value;
    return value.substring(0, limit) + trail;
  }
}

// Usage: {{ description | truncate:100:'…' }}

// Chaining pipes
// {{ name | uppercase | truncate:20 }}

// Async pipe (auto subscribes AND unsubscribes)
// <div *ngIf="user$ | async as user">{{ user.name }}</div>
💡 Interview Tip: The async pipe is one of the most important Angular tools. It automatically subscribes to an Observable/Promise, updates the view when data arrives, and unsubscribes when the component is destroyed — preventing memory leaks automatically.
ℹ️ Pure vs Impure Pipes: Pure pipes (default) only run when input reference changes — very performant. Impure pipes (pure: false) run on every change detection cycle — use sparingly. Never use impure pipes for filtering large arrays — it kills performance.

Route Guards (Functional — Angular 14+)
// Functional guard (modern approach)
export const authGuard: CanActivateFn = (route, state) => {
  const authService = inject(AuthService);
  const router = inject(Router);

  if (authService.isLoggedIn()) {
    return true;
  }
  return router.createUrlTree(['/login'], {
    queryParams: { returnUrl: state.url }
  });
};

// canActivateChild — for child routes
// canDeactivate — prevent leaving unsaved forms
// canLoad — prevent lazy loading unauthorized modules
// resolve — pre-fetch data before component activates

// Resolver
export const userDataResolver: ResolveFn<User> = (route) => {
  const userId = route.paramMap.get('id')!;
  return inject(UserService).getUser(userId);
};

// Access resolved data in component
// this.route.snapshot.data['userData']
Navigation & Route Parameters
// Programmatic navigation
@Component({ standalone: true, template: '...' })
export class ProductComponent {
  private router = inject(Router);
  private route = inject(ActivatedRoute);

  ngOnInit() {
    // Snapshot (one-time)
    const id = this.route.snapshot.paramMap.get('id');

    // Observable (reacts to param changes)
    this.route.params.pipe(
      map(params => params['id']),
      switchMap(id => this.productService.getById(id))
    ).subscribe(product => this.product = product);
  }

  goToProduct(id: number) {
    this.router.navigate(['/products', id]);
    // With query params:
    this.router.navigate(['/products'], { queryParams: { page: 2, sort: 'price' } });
  }
}

HTTP Interceptors (Functional — Angular 15+)
// auth.interceptor.ts
export const authInterceptor: HttpInterceptorFn = (req, next) => {
  const authService = inject(AuthService);
  const token = authService.getToken();

  if (token) {
    const authReq = req.clone({
      headers: req.headers.set('Authorization', `Bearer ${token}`)
    });
    return next(authReq);
  }
  return next(req);
};

// error.interceptor.ts
export const errorInterceptor: HttpInterceptorFn = (req, next) => {
  return next(req).pipe(
    catchError((error: HttpErrorResponse) => {
      if (error.status === 401) {
        inject(AuthService).logout();
        inject(Router).navigate(['/login']);
      }
      return throwError(() => error);
    })
  );
};

// Register in bootstrap
provideHttpClient(withInterceptors([authInterceptor, errorInterceptor]))
Services & the Service Pattern
@Injectable({ providedIn: 'root' })
export class UserService {
  private http = inject(HttpClient);
  private apiUrl = inject(API_URL);
  
  // Cache with shareReplay
  users$ = this.http.get<User[]>(`${this.apiUrl}/users`).pipe(
    shareReplay(1)
  );
  
  getById(id: number): Observable<User> {
    return this.http.get<User>(`${this.apiUrl}/users/${id}`);
  }
  
  create(user: Partial<User>): Observable<User> {
    return this.http.post<User>(`${this.apiUrl}/users`, user);
  }
  
  update(id: number, changes: Partial<User>): Observable<User> {
    return this.http.patch<User>(`${this.apiUrl}/users/${id}`, changes);
  }
  
  delete(id: number): Observable<void> {
    return this.http.delete<void>(`${this.apiUrl}/users/${id}`);
  }
}
Chapter 10 — Security

Authentication & Authorization

Implementing secure authentication in Angular requires JWT tokens, route guards, HTTP interceptors, and careful token storage strategies.

JWTRoute GuardsOAuth2RBAC
JWT Authentication Flow
1

User Logs In

POST credentials to /api/auth/login. Server validates and returns { accessToken, refreshToken }.

2

Store Tokens

Store access token in memory (not localStorage!). Store refresh token in HttpOnly cookie for XSS protection. Never store tokens in localStorage in production.

3

Attach Token to Requests

Auth interceptor adds Authorization: Bearer token header to every outgoing request automatically.

4

Handle Token Expiry

On 401 response, use refresh token to get new access token. Queue failed requests and retry them. On refresh failure, logout user.

5

@Injectable({ providedIn: 'root' })
export class AuthService {
  private http = inject(HttpClient);
  private router = inject(Router);
  
  // Signal-based auth state
  private currentUser = signal<User | null>(null);
  isAuthenticated = computed(() => !!this.currentUser());
  userRole = computed(() => this.currentUser()?.role);

  login(credentials: LoginDto): Observable<AuthResponse> {
    return this.http.post<AuthResponse>('/api/auth/login', credentials)
      .pipe(tap(res => {
        this.storeTokens(res.accessToken);
        this.currentUser.set(res.user);
      }));
  }

  refreshToken(): Observable<string> {
    return this.http.post<{ accessToken: string }>('/api/auth/refresh', {})
      .pipe(map(r => r.accessToken));
  }
  
  hasRole(role: string): boolean {
    return this.userRole() === role;
  }

  logout() {
    this.currentUser.set(null);
    sessionStorage.removeItem('token');
    this.router.navigate(['/login']);
  }
}
JWT Basics
🔑

Header

Algorithm (HS256, RS256) and token type. Base64URL encoded JSON: {"alg":"HS256","typ":"JWT"}

📋

Payload (Claims)

Contains user data: sub (user id), iat (issued at), exp (expiration), role, etc. Not encrypted — don't store sensitive data!

✍️

Signature

HMACSHA256(base64(header) + "." + base64(payload), secret). Verifies token integrity. Cannot be forged without the secret.

⚠️ Security Warning: Never store JWTs in localStorage — it's vulnerable to XSS attacks. Store access tokens in memory (JS variable) and refresh tokens in HttpOnly cookies. Use Content Security Policy headers as an additional layer.

@Component({ standalone: true, template: `

      console.log('Count changed:', this.count());
    });
  }
}

Chapter 15 — Interview Prep

Interview Questions & Answers

120+ curated Angular interview questions from beginner to advanced. Click any question to reveal the detailed answer.

BeginnerIntermediateAdvancedSystem Design
🟢 Beginner Level
QWhat is Angular and how is it different from AngularJS?+
Angular (v2+) is a complete rewrite in TypeScript with a component-based architecture, whereas AngularJS uses JavaScript with an MVC pattern. Angular uses a hierarchical dependency injection system, has better performance with the Ivy compiler, uses RxJS for reactivity, and is mobile-friendly. AngularJS uses two-way data binding via dirty checking which is slower at scale. They are effectively two different frameworks.
QWhat is a Component in Angular?+
A component is the basic building block of an Angular application. It consists of: (1) A TypeScript class decorated with @Component that defines the logic, (2) An HTML template that defines the view, (3) Optional CSS styles scoped to that component. Components control a piece of the UI and can communicate with other components via @Input/@Output decorators or services.
QWhat is the difference between constructor and ngOnInit?+
The constructor is a TypeScript/JavaScript concept called when the class is instantiated. It's for setting up dependency injection. ngOnInit is an Angular lifecycle hook called after Angular has set all the @Input properties. Always use ngOnInit for initialization logic, data fetching, and setting up subscriptions — not the constructor — because @Input values aren't available in the constructor.
QWhat are Angular decorators? Name the most common ones.+
Decorators are TypeScript functions prefixed with @ that add metadata to classes, methods, or properties. Key Angular decorators: @Component (marks a class as a component), @NgModule (module declaration), @Injectable (marks a class as injectable as a service), @Directive (attribute/structural directive), @Pipe (transforms data in templates), @Input (parent-to-child data), @Output (child-to-parent events), @ViewChild/@ContentChild (DOM/component references).
QWhat is Dependency Injection in Angular?+
DI is a design pattern where a class receives its dependencies from external sources rather than creating them. Angular has a hierarchical DI system: (1) Root injector — services with providedIn: 'root' are application-wide singletons. (2) Module injector — services provided in a module. (3) Component injector — services provided in a component's providers array. This makes code modular, testable (mock dependencies), and avoids tight coupling.
QWhat is the difference between *ngIf and [hidden]?+
*ngIf removes or adds the element from the DOM entirely. When false, the component is destroyed (ngOnDestroy fires). When true, a new component instance is created. [hidden] keeps the element in the DOM but sets display:none — the component stays alive. Use *ngIf for conditional rendering that saves memory. Use [hidden] when you need to preserve component state and toggling is frequent (like tabs).
🟡 Intermediate Level
QExplain Change Detection strategies in Angular. What is OnPush?+
Default: Angular checks every component on every event (click, HTTP, setTimeout). OnPush: Angular only checks a component when: (1) an @Input reference changes, (2) an event originates from the component or child, (3) the async pipe fires, or (4) ChangeDetectorRef.markForCheck() is called. OnPush requires immutable objects — mutating an array won't trigger CD, you need to create a new array reference. Always use OnPush with signals or observables for best performance.
QWhat is the difference between Subject, BehaviorSubject, and ReplaySubject?+
Subject: Multicasting observable. Late subscribers miss previous emissions. Good for events. BehaviorSubject: Requires an initial value. New subscribers immediately get the current value. Perfect for state (user auth state, current route). ReplaySubject(n): Replays the last N emissions to new subscribers. Useful when you need history. AsyncSubject: Only emits the last value when complete(). Use BehaviorSubject for most state management needs in Angular.
QWhat is the difference between switchMap, mergeMap, concatMap, and exhaustMap?+
switchMap: Cancels previous inner observable when a new value arrives. Best for HTTP search (type-ahead) where old requests should be cancelled. mergeMap: Runs all inner observables concurrently. Good for parallel HTTP requests. concatMap: Queues inner observables — completes one before starting next. Good for sequential uploads. exhaustMap: Ignores new values while inner observable is active. Perfect for preventing double-submit on button clicks.
QHow do you prevent memory leaks in Angular?+
1. async pipe — auto-unsubscribes when component is destroyed. 2. takeUntilDestroyed() — Angular 16+ automatically unsubscribes. 3. DestroyRef.onDestroy() — modern cleanup hook. 4. Manual unsubscribe() in ngOnDestroy. 5. Use take(1) for one-time subscriptions. 6. Never subscribe in constructor. Common leak sources: WebSocket connections, setInterval, event listeners, and route data subscriptions.
QWhat is a pure pipe vs impure pipe? When would you use each?+
Pure pipe (default): Angular only re-runs the transform when the input value reference changes. Very performant. Impure pipe (pure: false): Re-runs on every change detection cycle regardless of input changes. Use impure pipes for: filtering/sorting arrays when items change internally (mutation), pipes that depend on external state (like a translate pipe reading from a changing dictionary). Be careful — impure pipes can cause serious performance issues in large lists. Prefer recreating array references over impure pipes.
QWhat is lazy loading and how does it work in Angular?+
Lazy loading defers loading of feature modules/components until the user navigates to that route, reducing initial bundle size. In Angular 17+, use loadComponent: () => import('./path').then(m => m.Component) in routes. The router code-splits these into separate JS chunks loaded on demand. Use withPreloading(PreloadAllModules) to preload all lazy modules after initial load, or QuicklinkStrategy to preload only visible links.
QWhat is the ViewEncapsulation and its types?+
ViewEncapsulation.Emulated (default) — Angular adds unique attributes to CSS selectors, scoping styles to the component without true shadow DOM. ViewEncapsulation.ShadowDom — uses native browser Shadow DOM for true style encapsulation. ViewEncapsulation.None — styles become global, bleeding into child and parent components. Use Emulated for most cases, None when you intentionally want global styles, ShadowDom for web component compatibility.
🔴 Advanced Level
QWhat are Angular Signals and how do they replace RxJS?+
Signals are synchronous reactive primitives: signal() creates a writable signal, computed() creates a derived signal, and effect() runs side effects. Unlike RxJS Observables which are lazy and asynchronous, Signals are always synchronous and eagerly evaluated. Signals are better for local component state; RxJS is still superior for async streams, HTTP, and complex event composition. Angular 17+ supports toSignal() and toObservable() for bridging. Signals enable zoneless change detection — the future of Angular performance.
QExplain the NgRx store architecture and when to use it.+
NgRx implements Redux: a single immutable state tree managed by reducers, updated only through dispatched actions, with side effects handled by Effects. Use NgRx when: (1) Many components need the same data, (2) State changes are complex with many transitions, (3) Time-travel debugging is needed, (4) Team size is large and predictability matters. Don't use NgRx for simple state — component signals or a service with BehaviorSubject handles 80% of use cases with less boilerplate. NgRx Signals (@ngrx/signals) is the modern alternative with a simpler API.
QHow would you optimize a slow Angular application?+
1. OnPush change detection — biggest single win. 2. Lazy loading — reduce initial bundle. 3. trackBy in @for — prevent DOM recreation. 4. @defer — defer non-critical components. 5. Signals + Zoneless — remove Zone.js overhead. 6. Virtual scrolling (CDK) for large lists. 7. Production build with AOT compilation. 8. Bundle analysis with ng build --stats-json + source-map-explorer. 9. Pure pipes over computed values in templates. 10. Avoid function calls in templates — use getters or computed signals.
QWhat is Server-Side Rendering (SSR) in Angular and how does hydration work?+
Angular Universal / SSR renders the application on the Node.js server and sends pre-rendered HTML to the browser. This improves SEO and perceived performance. Hydration (Angular 16+) is the process of attaching event listeners and Angular's reactivity to the server-rendered HTML without re-rendering the DOM. Incremental hydration (Angular 19+) allows components to be hydrated lazily using @defer triggers — only hydrating what the user interacts with. This is the fastest Angular rendering approach for content-heavy sites.
QExplain the Angular Ivy compiler. What improvements did it bring?+
Ivy is Angular's rendering and compilation pipeline (default since Angular 9). Benefits: (1) Smaller bundles — locality principle, only includes code used, better tree-shaking. (2) Faster compilation — incremental compilation, components compile independently. (3) Better debugging — humanreadable generated code, better stack traces. (4) Standalone components — possible because Ivy compiles per-component without module context. (5) Lazy compilation — lazy components can compile on demand. Ivy's locality principle is what enables standalone components in Angular 17+.
QWhat is the difference between providedIn: 'root' and providing in a component?+
providedIn: 'root' registers the service in the root injector — a single singleton instance is shared across the entire app. This service is tree-shaken if not used. Providing in a component's providers: [MyService] creates a new service instance per component instance — this instance is destroyed when the component is destroyed. Child components share the parent's injected instance. Use component-level providers when you need instance-scoped state (e.g., a form management service that should reset when the form component is destroyed).
🎯 System Design Questions
QHow would you architect a large-scale Angular application for a team of 20 developers?+
1. Nx Monorepo — multiple apps sharing libraries. 2. Domain-driven feature modules — each team owns a domain. 3. Shared library — common UI components, utilities. 4. Core library — auth, logging, error handling. 5. Strict module boundaries with Nx enforce rules. 6. Standalone components throughout for tree-shaking. 7. NgRx Signals or NgRx Store for global state. 8. ESLint with Angular rules + Prettier. 9. Storybook for component docs. 10. GitHub Actions/Azure DevOps CI/CD with caching. 11. Lazy load every feature for bundle splitting. 12. OpenAPI generated clients for type-safe API calls.
QHow do you implement an authentication system in Angular with JWT and refresh tokens?+
1. Auth service with Signal-based state. 2. Auth interceptor attaches Bearer token to requests. 3. Error interceptor catches 401, calls refresh endpoint, queues failed requests, retries on success. Use BehaviorSubject to prevent multiple simultaneous refresh calls. 4. Store access token in memory (JS variable), refresh token in HttpOnly cookie. 5. canActivate guard checks auth state, redirects to login with returnUrl. 6. canActivateChild for role-based sub-routes. 7. On app init, use APP_INITIALIZER to attempt silent token refresh before rendering. 8. Implement token expiry check before requests using JWT exp claim.
